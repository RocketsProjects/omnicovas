# PART I — THE CONSTITUTION

## ⚖️ Section 0 — The Ten Laws of Operation (v4.0)

*Every module, every line of code, every architectural decision is subordinate to these laws. They are not guidelines.*

| # | Law | Core Mandate |
|---|-----|-------------|
| 1 | **Confirmation Gate** | AI suggests. Commander decides. Always. |
| 2 | **Legal Compliance Protocol** | Every rule, license, EULA, ToS, and law is absolute. No exceptions. |
| 3 | **API Respect Protocol** | Rate limits are hard constraints. Maintain courteous relationships. |
| 4 | **AI Provider Agnosticism** | Gemini default. Anything or nothing by choice. |
| 5 | **Zero Hallucination Doctrine** | If it isn't verified, it isn't said. |
| 6 | **Performance Priority** | Zero lag. Function before flair. Always. |
| 7 | **Telemetry Rigidity** | Telemetry defines reality. Nothing else does. |
| 8 | **Sovereignty & Transparency** | Commander's data stays local. The system is always auditable. |
| 9 | **Original Integration** | We build our own. We integrate compliantly. |
| 10 | **Unified Intelligence, Independent Operation** | Centralized suite. No critical dependencies. |

### Law 1 — Confirmation Gate
> **The AI will never perform, trigger, or initiate any in-game action without explicit commander confirmation.**

- AI operates in "suggest, never act" mode at all times
- Hard confirmation layer between every AI recommendation and executable output
- Voice acknowledgment, UI click, or bound keypress required
- No background automation affecting game state without commander awareness
- Cannot be overridden by settings, macros, or scripts

### Law 2 — Legal Compliance Protocol
> **Every applicable real-world law, license, End User License Agreement, Terms of Service, and copyright is an absolute constraint. We do not interpret flexibility into compliance. When in doubt, we stop and verify.**

- Applicable real-world laws (copyright, GDPR, local privacy laws)
- Frontier Developments' EULA and CAPI terms
- Every external service's Terms of Service
- All open source license obligations (ours and dependencies')
- Trademark respect and required disclaimers
- Proactive compliance verification
- When legal interpretation is uncertain, pause until verified
- Compliance is a feature, not an afterthought

### Law 3 — API Respect Protocol
> **All external service rate limits are hard constraints, never guidelines. We maintain courteous, proactive relationships with service maintainers.**

- Every rate limit enforced at client level
- Every service cached aggressively
- Every maintainer notified when integrating their service
- Formal approval where required (Frontier, Inara)
- Informal courtesy where appropriate (Spansh, Elite BGS, EDAstro)
- Graceful degradation — never retry storms

### Law 4 — AI Provider Agnosticism
> **Gemini is the default AI. The system is architected so any AI — cloud, local, or none — can be swapped in without changing core logic.**

- Core brain never calls AI providers directly
- All providers implement same `AIProvider` abstract interface
- Provider selection is a single user config entry
- `NullProvider` ensures full functionality with zero AI
- Four modes: Gemini Free (default) / Premium Cloud / Local LLM / No AI

### Law 5 — Zero Hallucination Doctrine
> **Every AI in this system is strictly bound by verified Elite Dangerous knowledge and project-supplied facts. Speculation is forbidden.**

- Every AI prompt grounded in structured system prompt with verified mechanics
- Knowledge Base is the AI's only source of truth
- Unverifiable responses flagged and suppressed
- AI must respond "I don't have verified data on that" rather than guess
- Knowledge Base version-tagged, maintained rigorously

### Law 6 — Performance Priority
> **OmniCOVAS exists to deliver precise, high-speed intelligence. Function and speed take absolute precedence over personality and flair. Zero lag between game event and system response.**

- Zero blocking I/O in core
- Latency budgets per event type (combat <100ms, navigation <200ms)
- Modular isolation — slow processes never cascade
- AI persona is cosmetic; core routing is untouchable
- Conversational padding is forbidden
- Performance regressions are bugs, not tradeoffs

### Law 7 — Telemetry Rigidity
> **The system's reality is defined solely by the hardware and software telemetry it receives.**

- No assumptions about game state — only what telemetry confirms
- Inferred states clearly labeled as such, never presented as fact
- Source priority order for conflict resolution (local journal always wins)
- Never fabricate state to fill data gaps

### Law 8 — Sovereignty & Transparency
> **The commander owns their data. The system explains itself at all times. Data stays local unless the commander says otherwise. Every action, recommendation, and outbound byte is visible and accountable.**

- Local-first by default
- Explicit opt-in for all outbound data
- Most restrictive privacy setting always wins
- Data flow transparency map visible in UI
- Every AI recommendation traceable to source
- Every API call logged with destination and payload summary
- Activity log accessible to commander
- No silent operations

### Law 9 — Original Integration
> **We build our own Elite Dangerous-specific features. We do not bundle or redistribute third-party software. Where interoperation with established third-party tools is legally permitted and adds value, we integrate compliantly — accepting required external tools as architectural partners for specific feature domains.**

**Interpretation B (Locked):**
- Core OmniCOVAS functionality (dashboard, overlay, telemetry, data analysis, squadron sync) works **standalone** without external tools
- Voice I/O features **require EDDI and VoiceAttack** — commanders install these separately per their own licenses
- All ED-specific features built natively
- No bundling, embedding, or redistribution of third-party software
- Standard open source libraries permitted under compatible licenses
- Community data services consumed as published (EDDN, EDSM, Inara, etc.)
- Documented migration path if a required integration partner becomes unavailable

### Law 10 — Unified Intelligence, Independent Operation
> **Game intelligence is centralized in this suite. Fragmentation is a failure state. Critical dependencies on external tools are also a failure state — we work alongside them, never dependent on them for core functionality.**

- Single source of truth for game intelligence during a session
- No required external tool for core (non-voice) functionality
- Optional external tool integrations for user convenience
- System fully operates with only OmniCOVAS running (voice disabled)
- System integrates gracefully when users choose external tools
- Core dependencies forbidden for non-voice features; voice dependencies acceptable

## 🎯 Section 0.5 — The Ten Architectural Principles

*Principles guide implementation. They are not laws but shape every design decision.*

| # | Principle | Summary |
|---|-----------|---------|
| 1 | **Accessibility** | System usable by commanders with diverse abilities and setups |
| 2 | **Sustainability** | Project operates at zero ongoing cost, sustainable by volunteer effort |
| 3 | **Evolutionary** | System adapts to game updates, community changes, and user feedback |
| 4 | **Attribution** | Every data source, contributor, and dependency credited visibly |
| 5 | **Graceful Failure** | Every external dependency has a local fallback |
| 6 | **Knowledge Base Stewardship** | KB treated as critical infrastructure; every entry version-tagged and audited |
| 7 | **Privacy-by-Default** | Every default setting is the most privacy-protective option |
| 8 | **Defensive Update** | Game patches and tool updates validated; assumptions tested after every change |
| 9 | **Explainability** | Every AI recommendation traceable to source on demand |
| 10 | **Resource Efficiency** | Memory, CPU, disk, and bandwidth use measured, bounded, and transparent |

> **Event-driven. Async-first. Modular by design. Local-first for latency. Law-bound by constitution. Zero-budget by design. Built for the long term.**

OmniCOVAS is a single, portable, high-performance intelligence suite for Elite Dangerous. It replaces the fragmented ecosystem of third-party tools with one unified command deck — governed by the Ten Laws, shaped by the Ten Principles, and architected for zero-latency response to live game telemetry. The project operates at zero ongoing cost, sustained by volunteer effort and community trust.

## 📡 Section 2 — Data Pipeline (Layered by Latency)

| Priority | Source | Latency | Purpose |
|----------|--------|---------|---------|
| 1st | **Journal Files** (local) | ~0ms | Raw game events, fastest possible |
| 2nd | **Status.json** (local) | ~0ms | Live pip/heat/flag state |
| 3rd | **EDDI** (optional, voice only) | ~10-50ms | Voice event triggers |
| 4th | **Local SQLite (bulk dumps)** | ~1-10ms | Galaxy-wide data pre-loaded |
| 5th | **CAPI** (Frontier, gated) | ~100-500ms | Ship loadouts, carriers |
| 6th | **EDDN** (live feed) | ~200-800ms | Crowdsourced data |
| 7th | **EDSM / Inara / EDAstro / Spansh / Elite BGS** | ~500ms-2s | Supplementary data |

**Key Principle:** Native journal parsing is primary. EDDI is the voice bridge, not the data source.

### UI Framework — Tauri
- Binary size: 3-10 MB
- Native OS WebView transparency for overlay
- Modern web stack frontend
- Rust backend performance
- Clean separation: Python intelligence, Tauri UI, bridged via FastAPI

### Python Stack
- `Python 3.11+` — core brain
- `asyncio` — async event loop
- `watchdog` — journal file watcher (native)
- `websockets` — EDDN live feed
- `aiohttp` — async API clients
- `SQLite + SQLAlchemy` — local state, caching, bulk data
- `FastAPI` — internal REST bridge to Tauri
- `cryptography` — local encrypted config vault
- `structlog` — structured JSON logging

### Voice Stack (Optional Feature Requirement)
- `EDDI` — required for voice I/O features (user-installed, external)
- `VoiceAttack` — required for voice command processing (user-installed, external, commercial)
- Python ↔ EDDI via localhost TCP socket bridge
- Core OmniCOVAS works fully without voice enabled

### P2P Networking Stack
- `ChaCha20-Poly1305` — authenticated encryption
- STUN (Google's free public servers) — primary
- Peer-Relay Fallback — squadron members volunteer (opt-in)
- Zero infrastructure cost

## 🤖 Section 5 — AI Abstraction Layer

```python
class AIProvider:
    async def query(self, prompt: str, context: dict) -> str:
        raise NotImplementedError

class GeminiProvider(AIProvider):      # Default
class OpenAIProvider(AIProvider):      # Optional cloud
class OllamaProvider(AIProvider):      # Local LLM
class NullProvider(AIProvider):        # No AI mode
    async def query(self, prompt, context):
        return None
```

### The Four AI Operating Modes

| Mode | Description |
|------|-------------|
| **Gemini Free Tier (Default)** | Aggressive caching, batched queries, stays in free tier |
| **Premium Cloud** | OpenAI/Claude/paid Gemini — user pays |
| **Local LLM** | Ollama — runs on-device |
| **No AI (NullProvider)** | Full functionality, pure telemetry |

### Three AI Integration Tiers

- **Tier 1 — Pure Telemetry** — Deterministic lookups. Works everywhere.
- **Tier 2 — Light AI (Rule-Based)** — KB rules, no LLM. Works everywhere.
- **Tier 3 — Full AI (LLM-Dependent)** — Natural language. Requires AI mode.

### Explainability Implementation (Tier-Based)

- **All tiers:** Logged in Activity Log with full traceability
- **Tier 3 responses:** Inline "Why?" button next to each recommendation
- **Future v1.1:** Expand inline buttons to Tier 2 based on user demand

### Top Secret Mode AI Behavior

When Top Secret Mode active, AI provider forced to `NullProvider` regardless of user setting. Commander sees:

> *"AI queries disabled during Top Secret operations to prevent context leakage. Configure Local LLM mode for AI during covert ops."*

### Knowledge Base Stewardship (Principle 6)

```
/omnicovas/knowledge_base

  exploration_mechanics.json
```

**Every entry has:**
- `patch_verified` — last ED game version the entry was verified against
- `source` — where the information came from
- `last_updated` — ISO timestamp
- `confidence` — high / medium / low
- `needs_review` — boolean flag

**KB Validation Pipeline (Phase 1 deliverable):**
- Automated tests verify entries match journal data patterns
- On game version detection, all entries flagged for review
- Pull request template for community contributions
- Audit dashboard shows KB health at a glance

## 🔌 Section 6 — EDDI Integration Strategy (Voice Only)

Under Law 9 Interpretation B, EDDI is **required for voice features**, not for core operation.

```
Game Journal Files + Status.json
      ↓ (PRIMARY DATA PATH — native parsing)
  OmniCOVAS Native Journal Parser
      ↓
  Core Brain (asyncio event dispatcher)
      ↓
  Confirmation Gate → Module → Response
      ↓                          ↓
  UI / Overlay          Optional Voice Output
                              ↑
                  (via EDDI + VoiceAttack bridge,
                   only if voice features enabled)
```

**Voice features require:**
- EDDI installed separately by commander
- VoiceAttack installed separately by commander (commercial ~$10-15)

**Without voice:**
- Full OmniCOVAS functionality via overlay + keyboard/mouse
- All pillars work
- All telemetry works
- All squadron features work

**OmniCOVAS provides:**
- Recommended EDDI configuration profile
- Pre-built VoiceAttack profile for OmniCOVAS commands
- Setup wizard for voice features (guides user through EDDI/VoiceAttack install)
- Documentation on the voice feature dependency

## 💰 Section 7 — Licensing & Future Monetization

### License: AGPL-3.0 with Contributor License Agreement (CLA)

**Why AGPL-3.0:**
- Copyleft protects against proprietary forks
- Network-use clause closes SaaS loophole
- Protects creator's commercial rights
- Allows personal and community use
- Compatible with GPL-3.0 community tools (EDDI via socket)

**Why CLA:**
- All contributors grant project creator license to their contributions
- Preserves future dual-licensing option
- Standard practice (Qt, MongoDB, Django all use CLAs)
- Contributors retain their own copyright and can use their code elsewhere
- ~30 minutes to implement using Apache ICLA template

**Commercial Paths (Preserved):**
- Donations (GitHub Sponsors, Ko-fi, Patreon)
- Dual licensing (AGPL free + commercial license for businesses)
- Paid services around open source core (premium AI integrations, cloud backup — future)

| Dependency | Fallback |
|-----------|----------|
| EDDN stream | Local cache |
| EDSM API | Local cache + Spansh dumps |
| Inara API | Local cache (extended) |
| Frontier CAPI | Journal data |
| Spansh API | Bulk dumps |
| EDAstro API | Local cache |
| Elite BGS API | Local historical snapshot |
| BGS-Tally | Optional (no fallback needed) |
| EDDI (voice) | Disable voice features, keep core |
| VoiceAttack (voice) | Disable voice features, keep core |
| AI Provider | NullProvider |

**System never hard crashes. Degraded mode always available.**
