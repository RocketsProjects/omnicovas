--------------------------------------------------------------------------------
ARCHITECTURE PATTERNS FOR PHASE 2
--------------------------------------------------------------------------------

Pattern 1 — Event Broadcasting (Publish/Subscribe)
  - Broadcast is async fire-and-forget
  - One slow subscriber must not block others
  - Subscriber exceptions caught in _safe_dispatch — never crash the publisher
  - Task references held in a set to prevent premature garbage collection
  - Event types are string constants in omnicovas/core/event_types.py

Pattern 2 — Latency Budget Enforcement
  - Every event type has a latency budget
  - Dispatcher measures elapsed time and logs warnings on breach
  - CI hard-fail with 2x tolerance multiplier for runner variance
  - Local budgets: HullDamage <100ms, ShieldsDown <100ms, FSDJump <200ms,
    Docked <200ms, Loadout <500ms

Pattern 3 — StateManager Extension (No Parallel State)
  - Extend the existing StateManager dataclass — never create ShipStateManager
  - Two state objects = two sources of truth = Law 7 violation
  - Source priority (journal > status_json > capi > eddn) applies to all new fields

Pattern 4 — KB-Grounded Analysis
  - Every judgment references a KB entry, not a magic number
  - kb.lookup("combat_mechanics", "hull_critical_threshold") is verifiable
  - Every KB entry: patch_verified, source, last_updated, confidence,
    needs_review, _justification

Pattern 5 — Tier-Aware Feature Implementation
  - Tier 1 (Pure Telemetry): Rebuy Calculator, Power Distribution
  - Tier 2 (Rule-Based): Heat Management — KB rules, no LLM
  - Tier 3 (LLM-Dependent): FORBIDDEN in Phase 2
  - Every feature works in NullProvider mode

--------------------------------------------------------------------------------
SEVEN-LAYER DEBUGGING VOCABULARY
--------------------------------------------------------------------------------

Trace in this order. Bugs hide in transitions between layers.

1. Watcher     — watchdog sees file change. Does logger.debug fire in-game?
2. Dispatcher  — routes event type to handlers. Is the event type registered?
3. Handler     — feature logic. Is it registered? Does it crash silently?
4. State       — StateManager updates. Is source_priority blocking? Nullable?
5. Broadcaster — pub/sub dispatch. Is event type a registered string constant?
6. Subscriber  — handler reacting to broadcast. Registered before first fire?
7. KB          — grounding. Entry present? Schema-valid? _justification set?
