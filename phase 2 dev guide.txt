How to Use This Document

This document is your step-by-step guide through Phase 2 of OmniCOVAS development. Phase 2 delivers the first user-facing pillar — Ship Telemetry — and the architectural backbone every future pillar will subscribe to. If Phase 1 was laying the concrete, Phase 2 is building the first floor.

🧭 The Golden Rule:  Never start a new week until every checkbox from the previous week is ticked. Each week's tasks depend on the previous week being complete. The Ship State Broadcaster in Week 7 is the foundation for every broadcast feature that follows — get it right before moving on.

You will be asking for help with nearly every task — that is expected and fine. When you ask, always tell your assistant:

• Which week and task you are on (e.g. "Week 7, Part A, ShipStateBroadcaster task 3")

Architecture Orientation for Phase 2

Five new patterns anchor Phase 2. Read this section before Week 7 begins. When something goes wrong later, you will come back here to figure out which pattern is involved.

Pattern 1 — Event Broadcasting (Publish/Subscribe)

Pillar 1 is the Foundation that publishes to every other pillar. The ShipStateBroadcaster is the first concrete pub/sub mechanism in OmniCOVAS. Key invariants:

• Broadcast is async fire-and-forget — one slow subscriber must not block others.

• Subscriber exceptions are caught inside a safe-dispatch wrapper — they never crash the publisher (Principle 5 — Graceful Failure).

• Task references are held in a set to prevent premature garbage collection.

• Event types are strings registered in a single source of truth (omnicovas/core/event_types.py).

Pattern 2 — Latency Budget Enforcement

Every event type has a latency budget. The dispatcher measures actual elapsed time and logs warnings on breach. In CI, budget violations are hard fails (with a 2x tolerance multiplier to account for shared-runner variance). Local budgets: HullDamage <100ms, ShieldsDown <100ms, FSDJump <200ms, Docked <200ms, Loadout <500ms. CI budgets are doubled.

Pattern 3 — StateManager Extension (No Parallel State)

Pillar 1 adds many new state fields — current_ship_type, hull_health, shield_up, fuel_current, jump_range_ly, modules, sys/eng/wep pips, heat_level, and more. These go into the existing StateManager dataclass from Phase 1. Never create a parallel ShipStateManager — that would give you two sources of truth. Source priority from Phase 1 (journal > status_json > capi > eddn) applies to every new field.

Pattern 4 — KB-Grounded Analysis

Every judgment — "hull critical," "heat dangerous," "rebuy expensive" — references a Knowledge Base entry, not a magic number. A magic number like 20.0 has no provenance. A KB lookup like kb.lookup("combat_mechanics", "hull_critical_threshold") is verifiable, versioned, and documented. Every new KB entry in Phase 2 carries patch_verified, source, last_updated, confidence, needs_review, and a new _justification comment explaining the value choice.

Pattern 5 — Tier-Aware Feature Implementation

Every feature explicitly declares its AI tier and degrades gracefully. Tier 1 (Pure Telemetry) features work everywhere — Rebuy Calculator, Power Distribution. Tier 2 (Light AI / Rule-Based) uses KB rules without LLMs — Heat Management. Tier 3 (LLM-Dependent) is forbidden in Phase 2. Every feature must work in NullProvider mode.

ℹ️ The Seven-Layer Debugging Vocabulary:  When a Phase 2 feature misbehaves, name the layer you are examining: (1) watcher, (2) dispatcher, (3) handler, (4) state, (5) broadcaster, (6) subscriber, (7) KB. "I am looking at the handler" is clearer than "let me check the code." Bugs hide in the transitions between layers more often than within a layer.

Create omnicovas/core/broadcaster.py with ShipStateBroadcaster class

Methods: subscribe(event_type, handler), publish(event_type, event), and internal _safe_dispatch(). Use asyncio.create_task for fire-and-forget dispatch. Hold task references in a set (self._tasks) to prevent GC. Attach done_callback to remove completed tasks.

☐

Implement per-subscriber error isolation

Principle 5 (Graceful Failure). One crashing subscriber must not cascade into silence for all others.

_safe_dispatch wraps each await in try/except Exception. On exception, logger.exception with subscriber.__name__. Never let a subscriber exception propagate to publish().

☐

Define ShipStateEvent dataclass

Typed events give mypy and subscribers a contract. Untyped dicts leak implicit assumptions across subscribers.

Use @dataclass(frozen=True). Fields: event_type (str), timestamp (datetime), payload (dict), source (str — 'journal' or 'status_json'). Every broadcast payload conforms to this shape.

☐

Wire ShipStateBroadcaster into the dispatcher

Broadcaster must be instantiated once and shared by every handler. Multiple instances = missed subscriptions.

Create a single broadcaster in main() alongside the StateManager. Pass it to the dispatcher. Every handler receives the same broadcaster reference. Do not create broadcasters inside handlers.

☐

Write pytest tests for broadcaster

The broadcaster is foundational. Silent bugs here poison every downstream feature.

Test: (1) subscribers receive published events, (2) multiple subscribers of the same event type all fire, (3) a subscriber raising an exception does not prevent other subscribers from firing, (4) slow subscriber does not block fast subscriber (use asyncio.sleep to simulate), (5) publish with no subscribers is a no-op.

Implement jump range calculation

☐

Implement fuel threshold detection

FuelLow (below 25%) and FuelCritical (below 10%) are broadcast events the UI and AI eventually react to.

After every fuel update, compute percentage = (fuel_current / fuel_capacity) * 100. If crosses below 25 and last state was above 25, broadcast FUEL_LOW. If crosses below 10, broadcast FUEL_CRITICAL. Do not broadcast every tick — only on threshold crossing. Thresholds come from KB entries (Part E).

☐

Write pytest tests for Fuel & Jump Range

Threshold-crossing logic is where off-by-one bugs live.

Test cases: fuel drops from 30% to 20% broadcasts FUEL_LOW once (not twice). Dropping from 15% to 8% broadcasts FUEL_CRITICAL once. Refueling from 5% to 80% does not re-broadcast anything. jump_range_ly populated from Loadout.

The Seven-Layer Debugging Vocabulary

When a Pillar 1 feature misbehaves, trace through these layers in order. Bugs hide in transitions between layers more often than within a layer.

Layer

What It Does / How to Check

1. Watcher

watchdog sees a journal file change. Check: does logger.debug in the watcher fire when you jump in game?

2. Dispatcher

Routes event type to handlers. Check: is the event type in your handler registration dict?

3. Handler

Feature-specific logic. Check: is it registered? Does it crash silently? (structlog exception log)

4. State

StateManager field updates. Check: is source_priority blocking the update? Is the field nullable?

5. Broadcaster

Pub/sub dispatch to subscribers. Check: is the event type a registered string constant? Are tasks being created?

6. Subscriber

Handler code reacting to broadcasts. Check: was the subscriber registered before the first broadcast fired?

7. KB

Knowledge Base grounding. Check: is the entry present? Schema-valid? Does _justification explain the value choice?

Architecture Decisions Locked in Phase 2

Decision

Why It Matters / Notes

ShipStateBroadcaster pub/sub

Every future pillar subscribes. Async fire-and-forget, per-subscriber error isolation, task references held to prevent GC.

Event types as string constants

Single source of truth in omnicovas/core/event_types.py. Prevents typos that silently break subscriptions.

StateManager extension, no parallel state

Adds fields to existing dataclass. Never a ShipStateManager. Source priority inherited from Phase 1.

Latency budgets as CI hard-fail

Matches mypy/ruff/pytest discipline. 2x tolerance multiplier for CI runner variance. Real regression catches real regressions.

Tier 1 and Tier 2 only

No LLM-dependent features in Phase 2. Everything works in NullProvider mode. Tier 3 waits for Phase 3+ when UI exists to show "Why?" inline.

KB-grounded thresholds

No magic numbers in Phase 2 handlers. Every threshold lives in KB with _justification. Makes changes reviewable and auditable.

jump_range_ly from Loadout.MaxJumpRange

Do not recompute from physics in Phase 2. Exploration pillar (Phase 5) owns first-principles math.

Feature modules one-per-file

omnicovas/features/{live_ship_state,fuel,hull_triggers,...}.py. Keeps Pillar 1 navigable. New contributors find code fast.

Integration test uses real recorded journal

Synthesized events miss edge cases. A real session captured to tests/fixtures/ is ground truth.

Critical Event Broadcaster is a contract, not a class

The six critical event types are the feature. A single audit test confirms they all fire. No new code class is needed.
