## 🗓️ Release Strategy

### v1.0 — "Core Intelligence" (Months 1-12)
**Goal:** Useful daily tool for commanders. Establishes architecture and trust.

**Scope:**
- Pillar 1 (Ship Telemetry) — complete
- Pillar 2 (Combat) — core features
- Pillar 3 (Exploration) — core features
- Pillar 5 (Trading/Mining) — core features
- Pillar 7 (Squadron) — basic P2P sync only
- UI shell with overlay
- Voice I/O (via EDDI/VoiceAttack) for basic queries
- Settings + privacy page
- First-run onboarding (Three-Path model)

### v1.1 — "Expansion" (Months 13-18)
**Goal:** Pillar completion and Powerplay/BGS.

**Scope:**
- Pillar 4 (BGS/PP2) — core features
- Pillar 6 (Engineering) — core features
- Pillar 2-5 feature additions
- Tier 2 inline Explainability UI
- Accessibility enhancements

### v1.2 — "Deep Features" (Months 19-24)
**Goal:** Advanced features, illegal ops, squadron stretch.

**Scope:**
- Full Illegal Ops suite
- Squadron stretch features
- Build Template Library
- Advanced mining/BGS features

### v1.5 — "Polish" (Months 25+)
**Goal:** Quality, performance optimization, remaining features.

### v2.0 — "Vision Complete" (Long-term)
**Goal:** All 166+ features. Native voice option. Advanced integrations.

# PART XI — DEPENDENCIES & BLOCKERS

## Critical Dependencies

| Dependency | Blocks | Mitigation |
|-----------|--------|-----------|
| Frontier CAPI approval | CAPI-dependent features in all pillars | Features gated behind approval, rest ship regardless |
| Inara whitelisting | CMDR lookups, faction data | Ship without, update when approved |
| EDDI (user install) | Voice features | Documentation clear; core features work without |
| VoiceAttack (user purchase) | Voice commands | Optional feature; documentation clear |
| Spansh data dump availability | Route plotting, system intel | If unavailable, fall back to live API (slower) |

## Known Risks

1. **Frontier CAPI approval denied or delayed** — ship v1.0 with Option C (features gated)
2. **Inara rate limit changes** — our architecture already tight against 2/min; resilient
3. **Game update breaks journal schema** — Defensive Update framework catches this
4. **Developer time pressure** — scope flexibility built in; features can defer

# PART XII — SUCCESS CRITERIA

## v1.0 is Successful If:

- ✅ Commanders can install and run OmniCOVAS on day one
- ✅ All five core pillars (1, 2, 3, 5, 7-basic) are functional
- ✅ Zero legal compliance issues
- ✅ Published on GitHub with AGPL-3.0 + CLA
- ✅ At least one positive review from an ED community member
- ✅ Frontier application submitted (approved or pending)
- ✅ Inara whitelisting requested
- ✅ Documentation sufficient for new commanders to succeed

## v1.0 is NOT Required to:

- ❌ Have all 166+ features implemented
- ❌ Have native voice (EDDI required is acceptable)
- ❌ Have advanced AI reasoning beyond basic queries
- ❌ Have all pillars complete
- ❌ Have commercial licensing set up
- ❌ Have bug bounty program
- ❌ Have paid code signing certificate
