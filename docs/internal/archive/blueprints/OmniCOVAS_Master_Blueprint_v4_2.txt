PART I — THE CONSTITUTION
================================================================================


--------------------------------------------------------------------------------
SECTION 1 — CORE PHILOSOPHY
--------------------------------------------------------------------------------

Event-driven. Async-first. Modular by design. Local-first for latency.
Law-bound by constitution. Zero-budget by design. Built for the long term.

OmniCOVAS is a single, portable, high-performance intelligence suite for Elite
Dangerous. It replaces the fragmented ecosystem of third-party tools with one
unified command deck, governed by the Ten Laws, shaped by the Ten Principles,
and architected for zero-latency response to live game telemetry. The project
operates at zero ongoing cost, sustained by volunteer effort and community
trust.


--------------------------------------------------------------------------------
SECTION 2 — THE TEN LAWS OF OPERATION
--------------------------------------------------------------------------------

Every module, every line of code, every architectural decision is subordinate
to these laws. They are not guidelines.

Summary table (in list form):

   1. Confirmation Gate. AI suggests. Commander decides. Always.
   2. Legal Compliance Protocol. Every rule, license, EULA, ToS, and law is
      absolute. No exceptions.
   3. API Respect Protocol. Rate limits are hard constraints. Maintain
      courteous relationships.
   4. AI Provider Agnosticism. Gemini default. Anything or nothing by choice.
   5. Zero Hallucination Doctrine. If it isn't verified, it isn't said.
   6. Performance Priority. Zero lag. Function before flair. Always.
   7. Telemetry Rigidity. Telemetry defines reality. Nothing else does.
   8. Sovereignty and Transparency. Commander's data stays local. The system
      is always auditable.
   9. Original Integration. We build our own. We integrate compliantly.
  10. Unified Intelligence, Independent Operation. Centralized suite. No
      critical dependencies.


Law 1 — Confirmation Gate

Mandate: The AI will never perform, trigger, or initiate any in-game action
without explicit commander confirmation.

   - AI operates in "suggest, never act" mode at all times.
   - Hard confirmation layer between every AI recommendation and executable
     output.
   - Voice acknowledgment, UI click, or bound keypress required.
   - No background automation affecting game state without commander awareness.
   - Cannot be overridden by settings, macros, or scripts.

Phase 2 application: Pillar 1 is primarily read-only telemetry broadcasting.
The few features that generate AI-style recommendations (Rebuy Calculator
advice, Heat Management suggestions) must pass through ConfirmationGate before
reaching the commander. Most Pillar 1 features are pure telemetry broadcasts
and do not invoke the Gate.


Law 2 — Legal Compliance Protocol

Mandate: Every applicable real-world law, license, End User License Agreement,
Terms of Service, and copyright is an absolute constraint. We do not
interpret flexibility into compliance. When in doubt, we stop and verify.

Phase 2 application: Pillar 1 operates entirely on local journal files and
Status.json data. No new external API calls are introduced in Phase 2.
Compliance posture is unchanged from Phase 1.


Law 3 — API Respect Protocol

Mandate: Rate limits are hard constraints. Maintain courteous relationships
with every external service maintainer.

Phase 2 application: No new external APIs. Rate Limit Registry unchanged.


Law 4 — AI Provider Agnosticism

Mandate: Gemini default. Anything or nothing by choice. Core brain never calls
AI providers directly; all providers implement a common interface.

Phase 2 application: Pillar 1 features default to Tier 1 (Pure Telemetry)
wherever possible. NullProvider must deliver full Pillar 1 functionality. Only
Heat Management and advanced Rebuy scenarios use Tier 2 (Light AI /
Rule-Based). No Tier 3 (LLM-dependent) features in Phase 2.


Law 5 — Zero Hallucination Doctrine

Mandate: If it isn't verified, it isn't said. Every AI prompt must be grounded
in Knowledge Base entries. Free-form knowledge is forbidden.

Phase 2 application: Every hull, module, or shield analysis references
verified KB entries. Any predictive output must be labeled with a confidence
score and never presented as certainty. "Hull at 18%" is telemetry; "Your
hull will fail in ~3 hits" is a prediction and must be marked as such.


Law 6 — Performance Priority

Mandate: Zero lag. Function before flair. Always.

   - No blocking I/O in the core event loop.
   - Local journal is the primary source (native parser).
   - In-memory state cache for zero-latency reads.
   - Debounced API calls.
   - Modular isolation so one slow subscriber cannot block the broadcaster.

Phase 2 application: Latency budgets are enforced and measured for the first
time. Combat events (HullDamage, ShieldsDown) must dispatch in under 100 ms.
Navigation events (FSDJump) must dispatch in under 200 ms. Every Pillar 1
event type has a budget registered in the dispatcher. Budget violations are
logged as warnings and surface in the Activity Log, and CI hard-fails on
breach with a 2x tolerance multiplier (see Section 14).


Law 7 — Telemetry Rigidity

Mandate: Telemetry defines reality. Nothing else does. Source priority is
enforced; local journal is ultimate truth. Missing fields stay missing until
real telemetry arrives.

Source priority (lowest number is highest priority):
   1. Journal (local, authoritative game events)
   2. Status.json (local, live hardware/UI state)
   3. CAPI (remote, Frontier API)
   4. EDDN (remote, crowdsourced)
   5. Inferred (OmniCOVAS-derived, never overrides real telemetry)

Phase 2 application: Pillar 1 IS the telemetry layer. Everything it publishes
must be derived from the journal or Status.json. Inferred state (for example,
"likely hot" before a ShipTargeted event fires) must be labeled as inferred,
never as fact.


Law 8 — Sovereignty and Transparency

Mandate: Commander's data stays local. The system is always auditable. Every
outbound data flow is opt-in. Every action the system takes is logged and
accessible.

Phase 2 application: Every Pillar 1 event is logged in the Activity Log.
Every broadcast to a subscriber is auditable. Squadron telemetry sync is
Phase 7, not Phase 2. Pillar 1 data stays local.


Law 9 — Original Integration

Mandate: We build our own. We integrate compliantly. EDDI/VoiceAttack remain
external. Core features work without them.

Interpretation B (locked in v4.0):
   - Voice features require EDDI (installed separately, GPL-3.0) and
     VoiceAttack (installed separately, commercial, approximately 10 to 15
     USD).
   - Core (non-voice) features must work entirely standalone.
   - OmniCOVAS does not bundle EDDI or VoiceAttack. Users install them
     under their own licenses.
   - Documented migration path if a required integration partner becomes
     unavailable.

Phase 2 application: Pillar 1 is fully native. No EDDI or VoiceAttack required
for any Pillar 1 feature. Voice output for ship-state queries is deferred
until Phase 3+ UI work is done.


Law 10 — Unified Intelligence, Independent Operation

Mandate: Game intelligence is centralized in this suite. Fragmentation is a
failure state. Critical dependencies on external tools are also a failure
state; we work alongside them, never dependent on them for core functionality.

   - Single source of truth for game intelligence during a session.
   - No required external tool for core (non-voice) functionality.
   - Optional external tool integrations for user convenience.
   - System fully operates with only OmniCOVAS running (voice disabled).
   - System integrates gracefully when users choose external tools.
   - Core dependencies forbidden for non-voice features; voice dependencies
     acceptable.

Phase 2 application: Pillar 1 is the first demonstration of the "unified but
independent" architecture. All Pillar 1 features must work with zero external
dependencies. Optional EDDN contribution (heat, fuel telemetry) is Phase 6+,
not Phase 2.


--------------------------------------------------------------------------------
SECTION 3 — THE TEN ARCHITECTURAL PRINCIPLES
--------------------------------------------------------------------------------

Principles guide implementation. They are not laws but shape every design
decision. Unchanged since v4.0.

   1. Accessibility — System usable by commanders with diverse abilities and
      setups. Keyboard navigation, high contrast modes, and font sizing in the
      UI phase.
   2. Sustainability — Project operates at zero ongoing cost, sustainable by
      volunteer effort. No subscriptions. Community-maintained.
   3. Evolutionary — System adapts to game updates, community changes, and
      user feedback. Defensive Update infrastructure supports this.
   4. Attribution — Every data source, contributor, and dependency credited
      visibly. SBOM generated per release. Contributors credited. Licenses
      visible.
   5. Graceful Failure — Every external dependency has a local fallback
      (EDSM to cache, Inara to cache, Spansh to bulk dumps, etc.).
   6. Knowledge Base Stewardship — KB treated as critical infrastructure;
      every entry version-tagged and audited. Validation pipeline runs in CI.
   7. Privacy-by-Default — Every default setting is the most
      privacy-protective option. Outbound data: OFF unless enabled.
   8. Defensive Update — Game patches and tool updates validated; assumptions
      tested after every change. Schemas checked.
   9. Explainability — Every AI recommendation traceable to source on demand.
  10. Resource Efficiency — Memory, CPU, disk, and bandwidth use measured,
      bounded, and transparent.

Principles specifically exercised by Phase 2:

   - Principle 5 (Graceful Failure): Pillar 1 subscribers degrade gracefully
     if Pillar 1 fails. A subscriber exception never crashes the broadcaster.
   - Principle 6 (KB Stewardship): Pillar 1 adds approximately 40 to 60 KB
     entries; all must be version-tagged (patch_verified, source,
     last_updated, confidence, needs_review) and audited by the existing
     Phase 1 validation pipeline.
   - Principle 9 (Explainability): Every Pillar 1 recommendation (Heat,
     Rebuy) includes a reference to the KB entry that grounds it. Logged in
     structlog. Accessible via Activity Log.
   - Principle 10 (Resource Efficiency): Pillar 1 is the first pillar that
     does sustained work; resource use must be measured via the existing
     ResourceMonitor and stay within resource_budget.yaml limits.

--------------------------------------------------------------------------------
SECTION 4 — DATA PIPELINE (LAYERED BY LATENCY)
--------------------------------------------------------------------------------

Seven data sources, ordered by latency and authority. Lower priority source
cannot override higher.

   1. Journal Files (local). Latency approximately 0 ms. Raw game events,
      fastest possible. Phase 1 status: operational.
   2. Status.json (local). Latency approximately 0 ms. Live pip, heat, and
      flag state. Phase 1 status: operational.
   3. EDDI (optional, voice only). Latency approximately 10 to 50 ms. Voice
      event triggers. Phase 3+ status.
   4. Local SQLite (bulk dumps). Latency approximately 1 to 10 ms.
      Galaxy-wide data pre-loaded. Phase 5 status.
   5. CAPI (Frontier, gated). Latency approximately 100 to 500 ms. Ship
      loadouts, carriers. Phase 3+ status, pending approval.
   6. EDDN (live feed). Latency approximately 200 to 800 ms. Crowdsourced
      data. Phase 5+ status.
   7. EDSM, Inara, EDAstro, Spansh, Elite BGS. Latency approximately
      500 ms to 2 s. Supplementary data. Phase 5+ status.

Key principle: Native journal parsing is primary. EDDI is the voice bridge,
not the data source.

SECTION 7 — AI ABSTRACTION LAYER
--------------------------------------------------------------------------------


A. Provider pattern

Every AI provider implements a common interface so swaps are a config change,
not a code change. Providers must never raise on typical failures; they log
and return None.

Concrete providers:

   1. GeminiProvider   — default cloud AI. Operational.
   2. OpenAIProvider   — optional cloud. Future.
   3. OllamaProvider   — local LLM. Future.
   4. NullProvider     — no AI mode. Operational.


B. Four AI operating modes

   1. Gemini Free Tier (Default)
      - Aggressive caching, batched queries, stays within the free tier.
      - Status: implemented.
   2. Premium Cloud
      - OpenAI, Claude, or paid Gemini. User pays.
      - Status: future.
   3. Local LLM
      - Ollama runs on-device.
      - Status: future.
   4. No AI (NullProvider)
      - Full functionality, pure telemetry.
      - Status: implemented.


C. Three AI integration tiers

   1. Tier 1 — Pure Telemetry. Deterministic lookups. Works everywhere.
      Most Pillar 1 features are Tier 1.
   2. Tier 2 — Light AI (Rule-Based). KB rules, no LLM. Works everywhere.
      Heat Management in Phase 2 is Tier 2.
   3. Tier 3 — Full AI (LLM-Dependent). Natural language. Requires AI mode.
      No Tier 3 features in Phase 2.


D. Explainability implementation

   - All tiers: logged in the Activity Log with full traceability (Phase 1
     operational).
   - Tier 3 responses: inline "Why?" button next to each recommendation
     (Phase 3 UI).
   - Future v1.1: expand inline buttons to Tier 2 based on user demand.


E. Top Secret Mode AI behavior (Phase 7)

When Top Secret Mode is active, the AI provider is forced to NullProvider
regardless of user setting. The commander sees:

    "AI queries disabled during Top Secret operations to prevent context
     leakage. Configure Local LLM mode for AI during covert ops."


F. Knowledge Base Stewardship (Principle 6)

Directory layout (/omnicovas/knowledge_base):

   1. combat_mechanics.json           — scaffold Phase 1, populated Phase 2.
   2. engineering_materials.json      — scaffold Phase 1.
   3. bgs_mechanics.json              — scaffold Phase 1.
   4. powerplay2_mechanics.json       — scaffold Phase 1.
   5. odyssey_mechanics.json          — scaffold Phase 1.
   6. trading_mechanics.json          — scaffold Phase 1, populated Phase 2.
   7. exploration_mechanics.json      — scaffold Phase 1.
   8. exobiology_mechanics.json       — scaffold Phase 1.
   9. api_limits.yaml                 — operational Phase 1.
  10. _metadata.json                  — KB-wide version and last audit date.

Every entry carries the following fields:

   - patch_verified    — last ED game version the entry was verified against.
   - source            — where the information came from.
   - last_updated      — ISO timestamp.
   - confidence        — high, medium, or low.
   - needs_review      — boolean flag.
   - _justification    — Phase 2 addition. Comment explaining value choice.

KB Validation Pipeline (operational from Phase 1):

   - Automated tests verify entries match expected schema.
   - On game version detection, entries are flagged for review.
   - Pull request template for community contributions.
   - Audit dashboard shows KB health at a glance (UI: Phase 3).

SECTION 8 — EDDI INTEGRATION STRATEGY (VOICE ONLY)
--------------------------------------------------------------------------------

Under Law 9 Interpretation B, EDDI is required for voice features and not
for core operation.

Behavior:

   - EDDI is never bundled. AGPL-3.0 + GPL-3.0 bundling conflict prevents it.
     Distribution remains as an external dependency that the user installs.
   - EDDI is only required when voice features are enabled by the user.
   - OmniCOVAS provides a recommended EDDI configuration profile and setup
     wizard.
   - If EDDI is not present, voice features are automatically unavailable and
     all other features continue working.

Phase 2 does not exercise this path. Full voice pipeline arrives in Phase 3+.


--------------------------------------------------------------------------------
SECTION 9 — LICENSING FRAMEWORK (AGPL-3.0 + CLA)
--------------------------------------------------------------------------------


A. Why AGPL-3.0

   1. Copyleft protects against proprietary forks.
   2. Network-use clause closes the SaaS loophole.
   3. Protects the creator's commercial rights.
   4. Allows personal and community use.
   5. Compatible with GPL-3.0 community tools via out-of-process integration
      (EDDI over socket).


B. Why a Contributor License Agreement (CLA)

   1. All contributors grant the project creator a license to their
      contributions.
   2. Preserves a future dual-licensing option.
   3. Standard practice (Qt, MongoDB, Django all use CLAs).
   4. Contributors retain their own copyright and can use their code
      elsewhere.
   5. Approximately 30 minutes to implement using the Apache ICLA template.


C. Operational status from Phase 1

   - LICENSE file (AGPL-3.0) committed.
   - CLA.md committed.
   - CLA Assistant integration documented (ready to enable when the first
     external PR arrives).
   - SECURITY.md committed with a responsible-disclosure policy.
   - CONTRIBUTING.md committed.


D. Commercial paths (preserved)

   - Donations: GitHub Sponsors (enabled), Ko-fi, Patreon.
   - Dual licensing: AGPL free plus commercial license for businesses.
   - Paid services around the open source core (premium AI integrations,
     cloud backup — future).

SECTION 14 — PERFORMANCE COMMITMENTS AND LATENCY BUDGETS
--------------------------------------------------------------------------------


A. Commitments

   1. No blocking I/O in the core.
   2. Local journal first (native parser).
   3. In-memory state cache.
   4. Debounced API calls.
   5. Modular isolation.


B. Latency budgets (enforced as CI hard-fail from Phase 2)

Format: event category — local budget (CI budget with 2x tolerance).

   1. Combat alerts (HullDamage, ShieldsDown):  under 100 ms (CI under 200 ms).
   2. Navigation events (FSDJump, Docked):       under 200 ms (CI under 400 ms).
   3. Trade queries (MarketBuy, MarketSell):     under 500 ms (CI under 1000 ms).
   4. Loadout events:                            under 500 ms (CI under 1000 ms).
   5. BGS analytics (Phase 8+):                  under 2 s    (CI under 4 s).


C. Enforcement mechanism (Python, summarized)

   - A BUDGETS dict maps event_type to a LatencyBudget(event_type, budget_ms).
   - dispatch_with_budget(event_type, event) wraps every dispatch, records
     elapsed_ms via time.perf_counter, and logs structlog warnings on breach.
   - BUDGET_MS is the local value; CI_BUDGET_MS = BUDGET_MS * (2 if CI else 1).
   - A dedicated test suite in tests/test_latency_budgets.py asserts
     elapsed_ms <= CI_BUDGET_MS for representative events, failing CI on
     breach.


D. Rationale for the CI tolerance multiplier

GitHub Actions shared runners have shown 2 to 3 times performance variance
versus dedicated hardware. A 2x tolerance prevents CI flakes without loosening
real discipline on local development.


E. Rationale for hard-fail versus warning

Law 6 is a law. Phase 1 enforced mypy, ruff, and pytest as hard fails.
Latency matches that discipline. Warning-only enforcement would let real
regressions ship.

   1. Detect current ED version from journal header.
   2. Compare to last known good version.
   3. Run schema validation tests for journal events.

SECTION 44 — WHAT WE LEARNED (WINDOWS AND TOOLING)
--------------------------------------------------------------------------------

   1. OneDrive conflicts with Git. Background sync, case-sensitivity quirks,
      and CRLF drift caused duplicate files (such as -TeamRocket.*) and
      git-status confusion after a folder move. Permanently disable OneDrive
      on project folders or use a non-synced path from the start.
   2. "ruff check --fix" is the correct invocation, not "ruff --fix".
   3. "UV_LINK_MODE=copy" is required on Windows when the uv cache and
      project cross filesystems or OneDrive folders. Set permanently with
      "setx UV_LINK_MODE copy".
   4. "git fetch --prune" should be configured globally
      ("git config --global fetch.prune true") so stale remote refs do not
      accumulate.
   5. Web-UI commits bypass pre-commit hooks. All code changes should go
      through local git, not GitHub's web editor.
   6. mypy type-checks untracked .py files in your package directory. The
      source-file count is a useful sanity indicator that cruft is gone.
   7. Test count is a proxy for feature count. 84 tests for Phase 1
      infrastructure. Phase 2 target is 130+ tests.
