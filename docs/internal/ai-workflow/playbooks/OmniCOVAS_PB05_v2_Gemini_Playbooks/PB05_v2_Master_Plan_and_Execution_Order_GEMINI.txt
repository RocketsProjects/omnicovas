GLOBAL PB05 v2 CONSTRAINTS
- Do not add dependencies unless the playbook explicitly opens a dependency gate. Default: no dependencies.
- Do not redesign backend FastAPI contracts unless explicitly allowed. Default: no backend contract changes.
- Do not build Phase 4 combat logic.
- Do not invent telemetry. Unknown, missing, unsupported, or unreliable telemetry stays truthful.
- Preserve Tauri/FastAPI dynamic bridge behavior.
- Preserve static vanilla Tauri UI architecture. No framework migration.
- Preserve classic script loading unless the playbook explicitly proves an existing pattern is safe.
- Use safe DOM rendering. Dynamic/API/user data uses textContent, createElement, replaceChildren, setAttribute.
- No unsafe dynamic innerHTML/outerHTML/insertAdjacentHTML.
- No Frontier assets, screenshots, traced silhouettes, extracted game models, or copied ship art.
- Ship schematics must be original internal SVG vector technical line art.
- Cool visuals are allowed only when they make the app easier to understand.

LOCKED PB05 v2 UX MODEL
- Left sidebar remains primary navigation.
- Sidebar labels remain simple: Dashboard, Activity Log, Resources, Settings, Privacy, Credits, Future Systems.
- Each route opens its own dedicated page.
- Dashboard is the only page with the ship wireframe / ship systems schematic hero.
- Other pages get route-specific identities but remain straightforward:
  Dashboard     -> Ship Systems Schematic
  Activity Log  -> Flight Recorder / Black Box
  Resources     -> Diagnostics Console
  Settings      -> Configuration Bay
  Privacy       -> Firewall / Data Control
  Onboarding    -> Commissioning Flow
  Overlay       -> HUD Projection
  Credits       -> Registry Plaque
- Dashboard schematic: top-down blueprint.
- First ship schematic: Sidewinder, recognizable but simplified original approximation.
- Generic fallback schematic is required.
- Dashboard panels: visible by default, hide/show from accessible hotspots.
- Panel hidden/shown state resets every launch for PB05.
- Future compact dashboard mode is architected for later, not implemented in PB05.
- Free-drag floating windows are forbidden in PB05. Snap zones are future optional work, not initial schematic MVP.

# PB05 v2 — Master Plan and Execution Order

STATUS / PROGRESS PRESERVATION
Already completed or mostly retained:
- PB05-PRE    Bridge-ready Settings/Privacy fix.
- PB05-PRE-B  Remaining fixed-port UI cleanup.
- PB05-00     Audit baseline.
- PB05-01     Tokens/Typography foundation.
- PB05-02     Shell/Nav/Future Systems.
- PB05-03     Internal SVG Icon System, pending PB05-03A correction if not committed.

Why v2 exists:
The original PB05 plan correctly established tokens, shell, icons, CSS extraction, dashboard summary, settings tabs, overlay settings, onboarding, accessibility, and readiness gates. PB05 v2 preserves those requirements but upgrades the remaining product direction from a generic card-grid dashboard to a route-based command deck with a Dashboard-only ship systems schematic.

PRIMARY DESIGN LAWS FOR PB05 v2
1. Left sidebar remains the primary navigation model.
2. Each route opens a dedicated page.
3. Dashboard alone uses the ship wireframe as the hero layout concept.
4. Other routes use themed page identities, but stay simple and readable.
5. No page may sacrifice usability for immersion.
6. Interactive ship/system controls live on Dashboard only.
7. Dashboard interactions open clear panels/callouts, not chaotic floating windows.
8. Settings stays tab-based and conventional in structure.
9. Privacy, Resources, and Activity Log are specialized consoles, not gimmicks.
10. Cool visuals are allowed only when they make the app easier to understand.
11. Orange means command/action/selection. Cyan/blue means telemetry/system data.
12. Purple is not part of the default PB05 palette.
13. All ship schematic art is original internal SVG vector line art only.
14. Missing telemetry gets truthful absence states.
15. Every interactive visual element must be keyboard accessible.
