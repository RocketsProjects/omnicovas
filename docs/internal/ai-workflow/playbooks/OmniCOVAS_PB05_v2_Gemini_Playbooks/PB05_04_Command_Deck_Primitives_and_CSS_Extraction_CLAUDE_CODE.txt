Phase C — Safe CSS extraction

1. Inspect ui/index.html for large inline <style> blocks and repeated inline
   style attributes.
2. Extract only low-risk repeated CSS into the new files.
3. Preserve all existing IDs and class names that controllers rely on.
4. If a style block is too large or risky to extract safely without behavior risk,
   leave it in place and document in the final report.
5. Do not rewrite route markup broadly.
6. Do not alter form control semantics.
7. Do not alter modal visibility behavior or DOM IDs.

Good extraction targets:
- repeated button styles
- repeated modal panel/backdrop styles
- repeated page heading/section styles
- repeated form row/label/control styles
- repeated chip/badge-like styles

Risky extraction targets to leave alone if uncertain:
- deeply nested onboarding wizard step styles
- privacy toggle switch mechanics
- any selector tightly coupled to current JS behavior
- any inline style that controls display/hide flow

Phase D — confirmation-gate.css token cleanup

1. Check ui/styles/confirmation-gate.css for undefined token names.
2. If any remain, replace with tokens defined in ui/styles/tokens.css.
3. Do not alter confirmation endpoint behavior or component JS.

Create ui/__tests__/index-style-structure.test.js if useful.
