HOW TO USE THIS DOCUMENT

This document is your step-by-step guide through Phase 4 of OmniCOVAS
development. Phase 4 implements Pillar 2: Tactical & Combat. If Phase 2 made
OmniCOVAS aware of the ship and Phase 3 made that awareness visible, Phase 4
turns the system into a combat-aware copilot: it listens to combat telemetry,
tracks targets and rewards, warns during interdictions, and surfaces tactical
context without ever taking control away from the Commander.

GOLDEN RULE: Phase 4 may advise, warn, explain, and display. It must never act.
Every combat advisory that could affect commander behavior must pass through the
Confirmation Gate surface before becoming a recommendation. Pure telemetry
updates may broadcast directly, but any "you should" output is a suggestion,
not an action.

THE THREE-COLUMN TASK TABLES

Every task appears with three columns plus a checkbox:
  Task           — What you are doing, in plain English.
  Why It Matters — The architectural reason.
  Notes          — Specific traps, warnings, and things to double-check.

ON GETTING HELP

When asking for help, always provide:
  - Which week and task (e.g. "Week 16, Part B, interdiction state task 3")
  - The exact error message
  - What you already tried
  - Which layer of the combat stack: watcher / dispatcher / handler / state /
    broadcaster / API endpoint / WebSocket / UI card / overlay / KB

--------------------------------------------------------------------------------
ARCHITECTURE PATTERNS FOR PHASE 4
--------------------------------------------------------------------------------

Pattern 1 — Combat Event Boundary
  - Extend the existing event taxonomy; do not create a second event bus.
  - Combat event constants live in the same event type source of truth used by
    Pillar 1.
  - Every combat feature publishes typed, auditable events.
  - Unknown journal fields remain unknown; do not fill with guesses.

Pattern 2 — Combat State Extends Session State
  - Add combat fields to the existing StateManager/session snapshot.
  - Do not create CombatStateManager.
  - Two combat state stores are a Law 7 violation.
  - Every state write declares TelemetrySource.

Pattern 3 — Target Snapshot Contract
  - Target Intelligence reads the current target as a snapshot.
  - A target snapshot may contain null fields.
  - It may say "unknown" but must not infer ship, rank, power, faction, or
    commander identity without telemetry or a verified local/external record.
  - The UI renders unknowns honestly.

Pattern 4 — Local-First PvP Intelligence
  - Local encounter history is the first source.
  - External CMDR lookup is optional, opt-in, and approval-gated.
  - Top Secret or privacy-restricted modes disable outbound lookup regardless
    of feature toggles.
  - Raw commander encounter logs stay local unless the user explicitly exports
    them.

Pattern 5 — Combat Advisory Gate
  - "Target is hostile" is telemetry when directly grounded.
  - "You should high-wake" is an advisory and must pass the Confirmation Gate
    surface.
  - Escape Planner outputs options, not actions.
  - No macro, keypress, or game action is triggered by Phase 4 code.

Pattern 6 — Overlay-First Critical Combat UX
  - Interdiction, hostile player, hull danger, and CZ-critical warnings belong
    in the overlay.
  - Combat history, reward tracking, and target details belong in the main UI.
  - Overlay banners must remain click-through by default.
  - Combat UI must not steal focus from Elite Dangerous.

Pattern 7 — KB-Grounded Threat Labels
  - Thresholds, classifications, and coaching labels come from the Knowledge
    Base.
  - If a threshold is not in the KB, the feature must either remain telemetry
    only or add a properly sourced KB entry.
  - Never invent combat mechanics or rank mappings in code.

Pattern 8 — Replay-Driven Combat Verification
  - Every combat feature needs journal replay fixtures.
  - Fixtures are architect-supplied or recorded from real sessions.
  - Soldier must never invent journal samples.
  - Integration tests replay target, interdiction, bounty, CZ, and PvP cases.

APPENDIX C — ESCALATION RULES
================================================================================

Escalate to Commander when any of the following occurs:
  1. A Playbook requires a journal fixture that is not attached.
  2. A task needs a combat threshold but no KB source is supplied.
  3. A task would add a new dependency.
  4. A task would call an external service without approval and opt-in gates.
  5. A task would automate an in-game action.
  6. Source priority blocks a write in a way the Playbook did not anticipate.
  7. Tests fail after the requested implementation.
  8. A v1.1/v1.2 feature appears necessary to complete a v1.0 task.
