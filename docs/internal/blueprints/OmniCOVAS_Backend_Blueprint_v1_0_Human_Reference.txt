Authority posture
-----------------
This document is authoritative for backend service boundaries, state ownership,
source routing execution, event models, workflow systems, API/bridge contracts,
cache and queue behavior, provenance, privacy enforcement at the service layer,
and backend implementation ownership.

It does not override:
- the Master Blueprint (laws, principles, pillars, phases, framework);
- the Compliance Matrix (legal, ToS, privacy, license, attribution, external);
- the UI Blueprint v1.0 (route ownership and user-surface design);
- the Source Capability Routing Reference (provider capability, request budgets,
  unsupported facts, fallback wording).

This document does override prior implicit backend assumptions in older planning
notes where those assumptions disagree with the locked UI Blueprint v1.0 route
ownership model or with the Source Capability Routing Reference v1 contract.

How to use this document
------------------------
Use this document when deciding:
- which backend service owns a piece of state, event, workflow, or API contract;
- whether a feature requires a new service or extends an existing one;
- which route consumes which backend output;
- which source is allowed to feed a given fact;
- what privacy, consent, authentication, cache, queue, budget, and Activity Log
  behavior must accompany a workflow.

If a question is about user-facing layout, route order, or which sidebar route a
feature lives under, use the UI Blueprint instead. If a question is about which
provider may answer a given fact and at what budget, use the Source Capability
Routing Reference instead. If a question is about laws, pillars, phases, or
constitutional scope, use the Master Blueprint instead.

2. EXECUTIVE SUMMARY
================================================================================

OmniCOVAS is event-driven, async-first, local-first, source-backed, auditable,
privacy-protective, AI-provider-agnostic, performance-first, and bound by
Confirmation Gate doctrine. The backend exists to make that posture real.

The backend owns:
- live local telemetry ingestion (journal watcher, Status.json reader, companion
  JSON readers);
- event dispatch and broadcast;
- canonical state (StateManager and its typed extensions);
- domain handlers and domain services (per pillar / per route family);
- the Knowledge Base and KB metadata;
- the source registry, source router, request queue, cache, and provenance
  registry (forward-looking surfaces);
- the workflow engine that turns commander intent into source plans, request
  bundles, recommendations, blockers, and next actions;
- the AI provider abstraction (NullProvider must always work);
- the Confirmation Gate;
- the Activity Log (canonical proof layer);
- the FastAPI bridge and the WebSocket event stream that feed the Tauri UI;
- the resource monitor, latency budget enforcement, and diagnostics surfaces;
- the DPAPI-backed secret vault and redaction processor.

The backend does not own:
- which sidebar routes exist (UI Blueprint owns route layout);
- which provider may answer which fact, at what budget, with which fallback
  wording (Source Capability Routing Reference owns this);
- legal, ToS, license, attribution, external relationship rules (Compliance
  Matrix owns these);
- pillars, phases, laws, principles (Master Blueprint owns these).

How this document connects to the others:
- UI Blueprint v1.0 defines what the commander sees. The Backend Blueprint
  defines what feeds it. Every UI route in the UI Blueprint has a backend
  ownership entry in Section 7 of this document.
- Master Blueprint defines the constitution. Every backend service in this
  document is bound by the Ten Laws (Master §2), Ten Architectural Principles
  (Master §3), and the Tier 1/2/3 AI doctrine (Master §7).
- Compliance Matrix defines the legal envelope. Every external workflow
  documented here cites the relevant Compliance Matrix section.
- Source Capability Routing Reference defines provider boundaries. Every
  ExternalRequestBundle in this document references the Source Capability
  Routing Reference Section 4 capability cards and the Section 6 budget
  recommendations rather than re-deriving them.
- Activity Log is the proof layer. Every backend service that mutates state,
  performs an external request, or runs through the Confirmation Gate writes
  to it.

3. BACKEND PHILOSOPHY
================================================================================

3.1 Local-first
The local Journal, Status.json, companion JSON files (Cargo.json, Market.json,
Outfitting.json, Shipyard.json, ModulesInfo.json, NavRoute.json), and the
OmniCOVAS state/cache are the primary truth path. External sources are
secondary. Missing telemetry stays missing. Inferred values must be labeled.

3.2 Event-driven
State changes arrive as events. The dispatcher routes events to typed handlers.
Handlers update StateManager and publish typed broadcasts. Subscribers consume
broadcasts asynchronously. There is one event taxonomy and one StateManager.

3.3 Async-first
No blocking I/O on the core event loop. External calls run on bounded async
workers. Slow or crashing subscribers are isolated by the broadcaster. Every
core path has a documented latency budget; CI enforces budgets where the active
phase guide requires it (Master §14).

3.4 Source-backed
Every fact carries source, timestamp, freshness, truth class, caveat, and an
Activity Log reference. AI never injects facts. Unknown remains unknown. There
is no synthesis of facts that no source supports.

3.5 Auditable
Every state mutation, every external request, every gate decision, every export,
every settings change, and every AI/voice/slash/route-activation decision lands
in the Activity Log with a stable event ID. The Activity Log is canon.

3.6 Privacy-protective
Outbound data defaults off. Secrets are encrypted at rest via DPAPI bound to the
commander's Windows account. Logs are redacted before write. No telemetry is
ever sent to project maintainers. Every outbound flow is consent-gated, opt-in,
and visible in the Data Flow / Sources surface in Settings.

3.7 AI-provider agnostic
The AIProvider interface abstracts every provider. NullProvider must always
work, and the core feature set must function with AI off. AI is never a source
of facts. AI cannot call APIs directly. AI cannot bypass the source router or
the Confirmation Gate.

3.8 Confirmation Gate
Any action that could affect the commander, the game, the local environment,
external services, or shared state passes the Confirmation Gate before it is
performed. The gate cannot be bypassed by AI, voice, slash command, route
activation, or workflow recommendation.

3.9 Performance-first
Resource Budget Framework (Master §11) and Latency Budget Framework (Master §14)
are constraints, not suggestions. The backend measures itself and surfaces its
own health through diagnostics rather than burying performance regressions.

4. AUTHORITY MODEL
================================================================================

Layer responsibilities:
- Master Blueprint v5.0 — constitutional umbrella, laws, principles, pillars, phases,
  release philosophy, authority model, and high-level doctrine.
- Compliance Matrix v4.1 — legal, ToS, privacy, licenses, external services,
  attributions, red flags.
- UI Blueprint v1.0 — frontend route ownership, route layout, user-surface
  containment, dashboard pin rules, alert lanes, slash command surface.
- Backend Blueprint v1.0 (this document) — backend services, state,
  source-routing execution, event models, workflows, API and bridge contracts,
  cache, queue, provenance, privacy enforcement at the service layer.
- Source Capability Routing Reference v1 — provider capability cards, allowed
  call types per workflow, request budgets, fallback wording, unsupported
  facts.
- Active phase guide / release notes — task detail and current shipped state.

Conflict handling:
- Master beats Backend Blueprint on laws, scope, phase, priority, doctrine.
- Compliance Matrix beats Backend Blueprint on legal, privacy, license, and
  external service rules. If a backend service appears to require a behavior the
  Compliance Matrix forbids, the service must be redesigned, blocked, or
  deferred.
- UI Blueprint beats Backend Blueprint on which routes exist and what each
  route owns at the surface. The Backend Blueprint may not invent a primary
  route, may not remove a route, and may not redesign route layout.
- Source Capability Routing Reference beats Backend Blueprint on which provider
  may answer which fact, at what budget, and with what fallback wording.
- Backend Blueprint beats lower-priority documents (CLAUDE.md, CLAUDE_CODE.md,
  GEMINI.md, individual playbooks) on backend service ownership, but is bound
  by all higher-priority documents above.

If a real conflict exists between two authorities, the conflict must be reported
and the higher authority obeyed until the constitution is updated.

  a WebSocket event stream consumed by the UI.
- Overlay window with critical alert lane (interdiction, hull critical) wired
  through the same broadcaster.
- First-Run Setup, Privacy page, Settings, Activity Log views, and Dashboard
  shell delivered.
- Output mode router covers UI text, voice draft, and overlay banner from a
  shared source.
- ADR 0003 locks the UI safe-rendering pattern (Tier 1 createElement and
  textContent preferred; Tier 2 escapeHtml + innerHTML allowed; Tier 3
  forbidden). Activity Log and overlay both follow the rule. Phase 3 closed at
  292 tests green.

5.5 Collision audit (forbidden duplications)
The following baseline systems already exist. Do not create a second one.
- Do not create a second StateManager, dispatcher, broadcaster, or event bus.
- Do not create a parallel cache layer for state already owned by StateManager.
- Do not create a second DPAPI vault.
- Do not create a second FastAPI bridge or a second WebSocket event stream.
- Do not create a second Activity Log or a second Confirmation Gate.
- Do not duplicate the AIProvider abstraction. New providers register into the
  existing factory.
- Do not create a parallel KB loader or KB validation pipeline.
- Do not create a "CombatStateManager" or any pillar-named StateManager. Combat,
  exploration, trade, mining, BGS, etc. extend the existing StateManager.

6. BACKEND LAYER STACK
================================================================================

The backend pipeline, top-to-bottom in event flow:

a. Local file watchers
   - Journal watcher: tails the latest Journal.*.log; handles rotation, mid-file
     restart, line buffering, encoding. Emits raw journal events.
   - Status.json reader: polls or watches Status.json safely; emits Status.json
     snapshots.
   - Companion JSON readers: read Cargo.json, Market.json, Outfitting.json,
     Shipyard.json, ModulesInfo.json, NavRoute.json on file change. Filter
     ModulesInfo.json against the current ship loadout slot set before mutating
     ship state to avoid SRV/on-foot context contamination.

b. Parser
   - Normalizes raw events into typed event records with timestamps and source
     classifications (journal vs Status.json vs companion JSON).

c. Dispatcher
   - Routes parsed events to typed domain handlers. Single source of truth for
     event-to-handler mapping. Lives in core; do not fork.

d. StateManager
   - Single canonical state object. Typed snapshots. Source priority rule:
     journal > Status.json > CAPI > EDDN > inferred. Domain handlers extend the
     snapshot through declared, typed fields, never through an ad-hoc dict.

e. Domain event handlers
   - Pillar-specific or route-family-specific handlers (Pillar 1 ship telemetry,
     combat target intelligence, interdiction, rewards, etc.). Each handler is
     small, named by responsibility, and registered with the dispatcher. The
     handler does not own state — it asks StateManager to mutate.

f. Domain services
   - Stable typed services that read StateManager and KB to answer route-driven
     questions (e.g., escape option generation, threat label lookup, market
     candidate review). Services are deterministic; AI may consume them.

g. SourceRegistry
   - Catalog of providers, their capability cards (per Source Capability
     Routing Reference Section 4), their current health, their auth state,
     their consent state, their rate-limit budget, their cache TTL defaults.
     The registry is the single point that translates "provider id" into
     "what may I do with it right now."

h. SourceRouter
   - Given a workflow and the set of facts it needs, the SourceRouter consults
     the SourceRegistry, the local-first sequence, and the Source Capability
     Routing Reference to produce a WorkflowSourcePlan. The router does not
     execute requests; it plans them.

i. WorkflowSourcePlan
   - Declarative plan stating, per fact: allowed sources, preferred order,
     cache key strategy, cache TTL, batch plan, rate-limit bucket, consent
     requirements, fallback behavior, unsupported fields, and the wording rules
     to use when a fact comes back missing or stale.

j. ExternalRequestBundle
   - One executable unit per provider per workflow run. Holds: workflow id,
     provider id, call type, endpoint family, fields requested, cache status,
     consent status, auth status, batch plan, queue policy, freshness
     requirement, fallback behavior, unsupported fields, status, and Activity
     Log references.

k. ExternalRequestQueue
   - Per-provider queue with bucket-aware scheduling, jittered backoff, hard
     caps, and graceful degradation when a provider returns 429 or is otherwise
     blocked.

l. Cache / freshness layer
   - Stores responses by cache key. Records source, endpoint, timestamp,
     freshness, TTL, source confidence, and stale behavior. Stale entries are
     served labeled, not silently. Cache hits and misses are auditable.

m. Provenance registry
   - Resolves "what is the source of this field right now" for any field shown
     in the UI. Required because facts may be a mix of live local telemetry,
     local snapshot, cached external, authenticated snapshot, community known,
     commander-entered, inferred, or AI draft.

n. Activity Log
   - Canonical proof layer. Receives events from every layer above and below.
     Stores commander-relevant view and full audit view. Every entry has a
     stable id; UI affordances such as Why and View Activity link by id.

o. Confirmation Gate
   - Wraps protected actions, including AI-suggested ones. Owns the proposal,
     confirmation, audit-id, and final state. Cannot be bypassed.

p. AI workflow planner
   - Translates commander intent into a draft WorkflowSourcePlan and a set of
     proposed steps. The planner does not produce facts. The planner output is
     advisory and labeled.

q. Workflow engine
   - Executes a plan: resolve cache, queue ExternalRequestBundles, gather
     responses, compute blockers, propose recommendations, surface next actions,
     hand protected actions to the Confirmation Gate, write to Activity Log.

r. FastAPI bridge
   - Serves snapshot endpoints to the Tauri UI and exposes the WebSocket event
     stream. Provides dynamic bridge/port discovery. Endpoint families are
     route-shaped (intel/*, navigation/*, operations/*, activity/*, settings/*,
     sources/*, diagnostics/*, about).

s. WebSocket event stream
   - Pushes typed events to the UI. Categories include telemetry, state
     updates, critical alerts, source events, workflow events, route events,
     audit events, overlay events, and (future) squadron events. Payloads carry
     source/freshness/truth/caveat fields where applicable.

t. Tauri UI contracts
   - The UI consumes the bridge URL discovered at startup; it does not embed a
     hard-coded localhost port. UI components apply the safe-rendering pattern
     (ADR 0003) and do not trust untrusted strings into innerHTML.

u. Diagnostics and Resource Monitor
   - Self-reports CPU, memory, latency budgets, source health, bridge health,
     cache health, queue health, KB health, and defensive update validation
     status. Surfaces through Settings → Diagnostics and through Activity Log
     errors/diagnostics.

7. ROUTE-TO-BACKEND OWNERSHIP MODEL (overview)
================================================================================

For each UI route in the UI Blueprint v1.0, there is one or more owning backend
service group. The route consumes summaries, snapshots, and event streams from
the owners. A route may consume from many services, but it does not own state
or workflow logic that another route owns. Sections 8–14 give the per-route
detail. Sections 23–24 give phase activation.

Mapping (one line per route):

- Dashboard backend: dashboard projection of StateManager snapshots, fixed-bar
  values, optional slot summaries, pin assembly, source/freshness chips, AI
  command bar status. Phase 1–3 baseline; extended per phase.
- Intel backend: known-information registry per Galaxy/System/Local/Ship-Personal
  scopes; field-level provenance; freshness; entity links; source lookup
  planning. Forward-looking new service group; Phase 5 begins it.
- Navigation backend: NavRoute.json proof, active route state, route candidates,
  planner jobs, route library, bookmarks, route source status. Forward-looking
  new service group; Phase 5 owns its scaffold; some bookmark/route concepts
  inherit from Phase 1–3.
- Operations backend: session focus, command board, active objectives, workflow
  state machines, recommendations, blockers, next actions, debriefs. Forward-
  looking new service group; Phase 4 introduces the first concrete operation
  packages (Combat).
- Activity Log backend: existing baseline (Phase 1+); extended by every later
  phase with new event families.
- Settings backend: existing baseline (Phase 1+, expanded in Phase 3); extended
  by every later phase with new categories such as Sources/Data Flow.
- About backend: existing baseline (Phase 3); extended only with new SBOM,
  attribution, and reporting-link content.
- Squadrons backend: forward-looking; activates in Phase 7.
- Engineering backend: forward-looking; activates in Phase 8.
- Carriers backend: forward-looking; reserved; activates when Fleet Carrier
  Full Command Center scope begins.

8. DASHBOARD BACKEND
================================================================================

Purpose
The Dashboard is the Live Command Surface. The Dashboard backend assembles a
compact projection from owning services so the UI can show what matters right
now without becoming a second source of truth.

Owns
- the fixed top core internal bar projection: Hull, Shields, Fuel, Heat, Pips
  (always present, always source-backed by live local telemetry from
  StateManager);
- the fixed bottom core external bar projection: Location, Status, Local,
  Objective, Data;
- the optional top internal slots projection (commander-selected from a
  candidate list);
- the left internal data rail projection: five commander-selectable summary
  cards drawn from internal-state services (cargo, modules, hardpoints,
  utilities, repair, materials, internal state, suit/SRV);
- the right external data rail projection: five commander-selectable summary
  cards drawn from external/contextual services (System Intel, Local Intel,
  Navigation, Operations, Trade/Market, BGS, Squadron, External Sources, future
  domain summaries);
- pin assembly for source-backed summary pins. Each pin record carries name,
  type, source, freshness, dashboard location, status (Active, Stale,
  Unavailable, Disabled, Commander-entered, Coming Later), origin route, and
  Activity Log reference;
- AI / command bar status and slash-command resolution for /open targets;
- output-mode routing of summary text for UI, overlay glance lane, and voice
  draft (where enabled), via the existing output-mode router.

Does not own
- known facts (Intel owns those);
- routes, planning, libraries, bookmarks, route sources (Navigation owns those);
- active workflow execution, workflow state, debriefs (Operations owns those);
- audit trails, source chains, external request records (Activity Log owns
  those);
- preferences, source setup, privacy controls (Settings owns those);
- license, security, contributing, CLA, SBOM (About owns those).

State models consumed (read-only)
- StateManager snapshot (typed).
- DashboardLayoutState (Settings owns persistence).
- Pin records (each owned by its origin service; Dashboard backend assembles).
- Source health summary (SourceRegistry).

Event families consumed
- state.* (live updates);
- intel.* (compact summaries only);
- operation.* (active objective summary only);
- route.* (active route summary only);
- source.* (data bar source health summary);
- gate.* (Confirmation Gate banners and pending decisions).

Endpoints (conceptual)
- GET dashboard state snapshot (full projection on demand).
- WebSocket subscription to typed events relevant to dashboard projections.

Privacy gates
- Dashboard never displays a value whose origin requires consent that has not
  been granted; the value is replaced with the appropriate fallback wording per
  Section 16.

Activity Log requirements
- Pin add/remove, layout change, optional slot change, AI command bar use, and
  slash command invocation are recorded.

Implementation phase
- Phase 1–3 baseline; per-phase extensions add new pin types and rail card
  candidates as new services activate.

Constraint: the Dashboard backend may never become a workflow board, route
planner, encyclopedia, audit console, or settings panel.

9. INTEL BACKEND
================================================================================

Purpose
Intel is the Known Information Console. The Intel backend is the canonical
service group for source-backed known information across Galaxy, System, Local,
and Ship/Personal scopes. Every fact carries source, timestamp, freshness,
truth class, caveat, and an Activity Log reference.

Owns
- Known Information Registry (KIR) — the catalog of known facts indexed by
  entity (galaxy/system/body/ring/station/carrier/settlement/site/POI/commander/
  ship/cargo/material/mission/discovery);
- field-level provenance for every fact;
- entity linking (e.g., a station belongs to a system; a body belongs to a
  system; a faction operates in systems and stations);
- source lookup planning hooks that hand off to the SourceRouter;
- caveat and truth-class labeling;
- "Send to Navigation," "Bookmark," "Attach to Operation," "View Activity,"
  "Queue source lookup" affordances backed by service calls;
- per-scope view assembly: Galaxy, System, Local, Ship/Personal.

Does not own
- routes or movement (Navigation owns those);
- workflow execution (Operations owns those);
- audit detail (Activity Log owns the deep view; Intel only carries activity
  references);
- settings or source configuration (Settings owns those).

State models (architectural contracts)
- KnownFact:
  - id, entity_ref, field_name, value_or_null, truth_class, source_id,
    source_endpoint_family, observed_at, ingested_at, freshness_state,
    cache_key, caveat, activity_log_ref, schema_version.
- EntityRef:
  - kind (galaxy / system / body / ring / station / carrier / settlement / site
    / poi / commander / ship / cargo / material / mission / discovery / fleet
    carrier), id, name (where available), parent_ref (where applicable),
    canonical_source.
- FieldProvenance:
  - source_id, endpoint_family, request_bundle_id (if external), cache_record_id
    (if cached), recorded_by ("local_journal" | "status_json" | "companion_json"
    | "capi" | "edsm" | "ardent" | "edastro" | "elitebgs" | "spansh" | "inara" |
    "eddn_listener" | "commander_entered" | "ai_draft" | "inferred"),
    recorded_at, expires_at_or_null, signature_or_null.
- FreshnessState:
  - one of {LIVE, FRESH, STALE, EXPIRED, UNKNOWN, NOT_LOADED}.
- TruthClass:
  - one of the values defined in Section 16.
- SourceCaveat:
  - one of {NONE, INCOMPLETE_COVERAGE, RATE_LIMITED, REQUIRES_AUTH,
    REQUIRES_CONSENT, COMMUNITY_OBSERVED, INFERRED, AI_DRAFT, UNSUPPORTED,
    NO_VERIFIED_SOURCE}; plus a free-text note where useful.
- IntelSnapshot:
  - scope (galaxy/system/local/ship-personal), entity_ref, facts_by_field,
    source_summary, last_updated, queued_lookups.

Event families
- intel.fact_recorded
- intel.fact_updated
- intel.fact_invalidated
- intel.lookup_queued
- intel.lookup_completed
- intel.lookup_blocked
- intel.scope_view_assembled
Each event payload carries source, freshness, and Activity Log reference.

Endpoints (conceptual)
- GET intel galaxy snapshot
- GET intel system <id>
- GET intel local <id>
- GET intel personal
- POST intel queue source lookup (creates a WorkflowSourcePlan via SourceRouter)
- GET intel fact <id> source chain (delegates to Activity Log)

Privacy gates
- Authenticated facts (CAPI) require consent and DPAPI-vaulted token.
- Inara-derived facts require whitelist plus consent.
- Commander-entered facts are local-only by default; no outbound.

Activity Log requirements
- Every fact_recorded / fact_updated event has a corresponding Activity Log
  entry. Source lookup requests, blocked attempts, and rate-limited responses
  are all logged.

Implementation phase
- Phase 5 stands up the first cross-cutting Intel service group; earlier phases
  already produce many of these facts under Pillar 1 / Phase 2 ship telemetry,
  but they are not yet exposed under a unified Intel surface. Phase 5 is the
  point at which the unified Intel scope assembly is required.

Constraint: Intel never produces an unsourced AI fact. Intel never claims a
fact it cannot defend with field-level provenance.

10. NAVIGATION BACKEND
================================================================================

Purpose
Navigation is the Route and Movement Console. The Navigation backend owns
everything related to where and how to move: NavRoute.json proof, active route
state, route candidates, planner jobs, route library, bookmarks, and route
source status.

Owns
- NavRoute.json ingestion (companion JSON reader);
- active route state (current destination, next jump, progress, route source);
- route candidate generation hooks (Spansh link-out today; future Spansh API/
  dump integration after terms verification; future internal candidate routers
  for trade loops, mining sweeps, BGS circuits);
- planner jobs (route intent, parameters, queue, status);
- route library (saved, recent, imported, expedition, exobiology, trade/mining,
  mission, engineering/material, carrier, BGS/system circuits, tags,

15. SOURCE ROUTING BACKEND
================================================================================

Purpose
The source routing backend ensures that every workflow needing external facts
asks the right provider, at the right budget, with the right consent and auth,
and falls back honestly when a fact is not available. It binds backend
implementation to the Source Capability Routing Reference v1.

Components
- SourceRegistry: catalog of providers and their capability cards (mirrors
  Source Capability Routing Reference Section 4). Tracks per-provider state:
  enabled/disabled, consent, auth, current bucket usage, last failure,
  attribution acknowledgment.
- ProviderCapability: typed record of which call types a provider supports,
  which fields each call returns, which call types it does NOT support, and
  which call types are forbidden.
- WorkflowSourcePlan: the plan-level contract; see Source Capability Routing
  Reference Section 6 and reproduce the field set:
  workflow_type, commander_intent, required_facts, optional_facts,
  allowed_sources_per_fact, preferred_source_order, cache_key_strategy,
  cache_ttl, request_bundle_plan, rate_limit_bucket, consent_requirements,
  fallback_behavior, unsupported_fields, output_wording_rules.
- ExternalRequestBundle: the executable unit; see Source Capability Routing
  Reference Section 6 and reproduce the field set:
  workflow_id, workflow_type, provider_id, call_type, endpoint_family,
  required_facts, optional_facts, fields_requested, cache_key, cache_status,
  source_priority, consent_status, auth_status, documented_provider_limit,
  documented_provider_guidance, omnicovas_bucket, batch_plan, max_batch_size,
  queue_policy, freshness_requirement, fallback_behavior, unsupported_fields,
  status, activity_log_ids.
- SourceRequest / SourceResponse: per-call records; HTTP details normalized;
  no secrets; status codes and rate-limit signals captured.
- CacheRecord: cache_key, provider_id, endpoint_family, response_summary
  (redacted), recorded_at, ttl, source_confidence, stale_behavior.
- RequestBudget: per-provider bucket implementing the recommendations in Source
  Capability Routing Reference Section 6; includes default/min/hard limits,
  current consumption, reset window, backoff state.
- SourceBlocker: typed blocker emitted to workflows when a request cannot
  proceed (kind = consent_missing / auth_missing / rate_limited /
  source_disabled / unsupported / needs_verification).
- UnsupportedFact: typed declaration that a fact is not supported by any
  enabled source; carries the recommended fallback wording.
- FieldProvenance: the provenance object attached to every fact (Section 9).
- ExternalRequestQueue: per-provider queue with bucket-aware scheduling,
  jittered backoff on 429, hard caps, graceful degradation.

Local-first sequence
For every workflow, the SourceRouter checks in this order:
1. Local Journal.
2. Local companion JSON files (Status.json, Cargo.json, Market.json,
   Outfitting.json, Shipyard.json, ModulesInfo.json, NavRoute.json).
3. OmniCOVAS state/cache.
4. Authenticated CAPI snapshot, if relevant and consent/auth granted.
5. Verified external sources per the Source Capability Routing Reference Section
   2 call-type router.
6. Link-only/manual sources or unsupported declaration if no verified source
   exists for a fact.

Source gates
- Consent: outbound calls default off; commander explicit opt-in required.
- Auth: tokens stored in DPAPI vault; never logged; required scope verified
  before send.
- Privacy: facts that risk exposing commander identity/location externally
  default off and are visible in the Data Flow surface.
- Attribution: provider-required attributions surfaced in About and on
  consuming surfaces.

Provider-specific boundaries (delegated)
- Per the Source Capability Routing Reference Section 2 call-type router,
  Section 3 provider priority map, Section 4 capability cards, and Section 6
  budget recommendations. The Backend Blueprint does not re-derive numbers; it
  binds the SourceRegistry to those values.

Implementation phase
- Phase 5 begins the SourceRegistry/SourceRouter scaffold (Intel + Navigation
  workflows). Phase 6 extends with Trade/Mining and Carrier needs. Phases 7–9
  add Squadrons, Engineering, and BGS/Powerplay needs.

Constraint: AI never bypasses the SourceRouter. All external calls flow through
the SourceRegistry → WorkflowSourcePlan → ExternalRequestBundle → queue path.

16. PROVENANCE AND TRUTH MODEL
================================================================================

Every field shown to the commander carries a TruthClass plus the field-level
provenance defined in Section 9. The TruthClass values are authoritative and
must not be substituted with alternate vocabulary.

TruthClass values
- LIVE_LOCAL_TELEMETRY — the value comes from the live Journal or Status.json
  reading.
- LOCAL_SNAPSHOT — the value comes from a companion JSON snapshot written by
  Elite Dangerous (Cargo.json, Market.json, Outfitting.json, Shipyard.json,
  ModulesInfo.json, NavRoute.json) or from a recorded local OmniCOVAS state.
- CACHED_EXTERNAL — the value comes from a previously fetched external response
  still within freshness; provenance includes provider, endpoint, recorded-at,
  and ttl.
- AUTHENTICATED_SNAPSHOT — the value comes from a CAPI authenticated endpoint;
  may not be live.
- COMMUNITY_KNOWN — the value comes from a community-sourced provider (EDSM,
  EDAstro, Ardent, EliteBGS, EDDN listener cache). Coverage is community-
  observed, not guaranteed.
- COMMANDER_ENTERED — the value was entered by the commander; treated as
  authoritative for the commander's local context but never as external truth.
- INFERRED — the value is derived from telemetry plus rules; must be labeled and
  confidence-scored; cannot override LIVE_LOCAL_TELEMETRY or LOCAL_SNAPSHOT.
- AI_DRAFT — the value is an AI-produced draft; cannot stand alone as a fact;
  must be accompanied by grounding refs or labeled clearly as a draft.
- UNKNOWN — no source has produced this value yet.
- NOT_LOADED — the source for this value exists but has not yet been read.
- DISABLED — the source is configured off.
- STALE — the cached value is past its freshness window; may be shown labeled.
- REQUIRES_AUTHORIZATION — the source needs commander authentication or
  consent that has not been granted.
- UNSUPPORTED — no enabled source supports this fact in this configuration.
- NO_VERIFIED_SOURCE — no source has verified support for this workflow yet
  (e.g., mining hotspot/ring candidate workflow today).

Field record contract
Every field record carries: source, source_endpoint_family, observed_at,
ingested_at, freshness_state, truth_class, caveat, activity_log_ref,
schema_version. The UI is required (per UI Blueprint v1.0 Section 19) to
display source/freshness chips, Why links, View Activity links, and the
fallback wording in Section 19 of this document.

17. WORKFLOW ENGINE BACKEND
================================================================================

Purpose
The workflow engine turns commander intent into a series of source-bounded,
gated, audited steps. It is the only place where commander intent becomes
external requests, and it never produces facts that no source supports.

End-to-end flow
1. Intent interpretation
   - Commander enters intent (typed in command bar, slash command, voice draft,
     or workflow start). The AI workflow planner may classify intent. Output is
     a candidate WorkflowSpec.
2. Source planning
   - SourceRouter produces a WorkflowSourcePlan per Section 15. Each required
     fact is mapped to an allowed source list and preferred order.
3. Source routing
   - The engine resolves cache first. For uncached facts, ExternalRequestBundles
     are queued. The provider queue enforces budgets and consent/auth gates.
4. Blocker detection
   - Consent missing, auth missing, rate-limited, source disabled, unsupported,
     and commander-input-required produce typed blockers visible to Operations.
5. Recommendation generation
   - Domain services compute recommendations grounded in StateManager and
     resolved facts. The AI workflow planner may produce drafts; drafts are
     labeled and tagged with grounding refs.
6. Confirmation requirements
   - Any protected action (route activation hint, settings change,
     export/delete, outbound submission, AI-suggested action with side
     effects) passes the Confirmation Gate (Section 19).
7. Next actions
   - The engine surfaces concrete next actions (commander_input,
     review_recommendation, copy_next_jump, open_intel, open_navigation,
     open_settings_source, etc.).
8. Activity Log linkage
   - Every transition writes to the Activity Log with a stable id; UI affordances
     resolve to those ids.

Constraints
- AI cannot create facts.
- AI cannot call APIs directly; AI proposes plans, the engine routes through
  the SourceRouter.
- AI cannot bypass the Confirmation Gate.
- The engine cannot perform a game action; in-game effects always require a
  commander action.

Phase mapping
- Phase 4 introduces the engine in narrow form (Combat operations).
- Phase 5 generalizes for Intel/Navigation/Exploration.
- Phase 6+ broadens scope by package.

19. CONFIRMATION GATE BACKEND
================================================================================

Purpose
The Confirmation Gate is the project-level guarantee that AI suggests and the
commander confirms. Every protected action passes through the gate.

Protected actions
- Settings changes that materially affect privacy, source enablement, voice,
  AI, or game controls.
- Outbound submissions (EDDN, BGS-Tally if enabled, Frontier-facing
  submissions, future shared squadron data).
- Exports and deletes of commander data.
- Route activation hints that translate into commander-initiated copy/paste of
  next-jump or destination data.
- AI-proposed actions with side effects.
- Future input assist (voice command apply, copy/paste actions).

States
- PROPOSED: the gate has a candidate action awaiting commander review.
- CONFIRMED: commander confirmed; the action is performed; an audit_id is
  emitted.
- CANCELED: commander declined; the action is not performed; audit_id is
  emitted.
- BLOCKED: a higher rule (privacy, compliance, source unavailability) prevents
  even prompting; the gate emits a BLOCKED record explaining why.

Action proposal model
- proposal_id, kind, target (settings_ref / source_ref / route_ref / intel_ref
  / operation_ref / export_ref / delete_ref), payload_summary, justification,
  source_refs, ai_draft_flag, severity, deadline_or_null.

Cross-surface relationships
- AI: every AI side-effect proposal is wrapped.
- Voice: voice apply requires an explicit command; the gate is still in front.
- Slash commands: deterministic shortcuts may not bypass protected actions.
- Route activation: copy/paste of next-jump is a commander action; the gate
  records the proposal and the commander's decision.
- External requests: high-impact submissions pass the gate even if a generic
  source toggle is on.
- Exports/deletes: always pass the gate; export and delete records are
  privacy-class events.
- Settings changes: high-impact toggles pass the gate; trivial preferences may
  not require it but are still logged.

Constraint: the Confirmation Gate cannot be disabled. AI, voice, and slash
commands cannot bypass it.

Posture
- Local-first by default.
- Outbound off by default.
- Secrets at rest encrypted via DPAPI bound to the commander's Windows account.
- Logs redacted at write time.
- No telemetry to project maintainers.

Privacy gates
- Per-source consent toggles.
- Per-action consent for outbound submissions.
- Sensitive Intel facts (commander encounter notes, PvP records) default to
  local-only; export requires Confirmation Gate.

Auth gates
- Tokens, OAuth, and API keys stored in vault. Scope checks before every send.
- Vault rotation supported and logged.

Source consent
- Provider toggles in Settings → Sources/Data Flow govern enablement.
- Attribution acknowledgments required before enabling certain sources.

Export and delete
- Commander can export their data and delete OmniCOVAS-held data via
  Settings → Privacy & Data. Confirmation Gate enforced. Activity Log records
  the operation; raw deleted data is not retained after a delete operation
  except for the audit metadata that proves it happened.

Squadron sharing privacy (Phase 7+)
- All shared telemetry is consent-gated and visible in the Data Flow surface.
- Top Secret Mode and Burn Command require future security doctrine before
  implementation; the backend documents them as RESERVED and gated.
- No shared route or shared bookmark synchronization happens without explicit
  share scope and Activity Log records.

Security reporting
- About → Security links to SECURITY.md and the project's vulnerability
  reporting URL. The backend itself never silently sends crash reports or
  diagnostics outbound; diagnostics surfaces are local.

21. BRIDGE / API CONTRACTS
================================================================================

The FastAPI bridge serves the Tauri UI. The bridge URL is discovered
dynamically at startup; the UI does not hard-code a localhost port. Endpoint
names below are conceptual unless the Phase 1–3 baseline has already named
them. New names appear only when the implementing phase activates.

Conceptual endpoint families
- Dashboard: GET dashboard state (full projection).
- Events: WebSocket subscription for typed events (telemetry, state, alerts,
  source, workflow, route, audit, overlay, future squadron).
- Intel: GET intel/<scope>, GET intel/fact/<id>, POST intel/queue-source-lookup.
- Navigation: GET navigation/active, GET navigation/candidates,
  POST navigation/plan, GET navigation/library, GET navigation/bookmarks,
  POST navigation/bookmark, GET navigation/sources, POST navigation/copy-next-jump.
- Operations: GET operations/command-board, GET operations/<id>,
  POST operations, POST operations/<id>/step, POST operations/<id>/recommendation,
  POST operations/<id>/debrief.
- Activity Log: GET activity/timeline, GET activity/event/<id>,
  GET activity/source-chain/<fact_id>, GET activity/external-requests,
  GET activity/ai-gates, POST activity/export, POST activity/clear.
- Settings: GET settings/<category>, POST settings/<category>,
  POST settings/reset, POST settings/sources/<provider>,
  POST settings/export, POST settings/import.
- Sources / queue: GET sources/registry, GET sources/queue,
  POST sources/<provider>/consent, POST sources/<provider>/auth.
- Diagnostics: GET diagnostics/health, GET diagnostics/budgets,
  GET diagnostics/cache, GET diagnostics/queue.
- About: GET about/metadata.

Endpoint discipline
- Bridge URL is dynamic; UI clients consume a discovered URL at runtime.
- All endpoints use typed payloads with explicit source/freshness/truth/caveat
  fields where applicable.
- All endpoints write to the Activity Log when they mutate state.
- POSTs that perform protected actions go through the Confirmation Gate.

================================================================================
22. EVENT STREAMING CONTRACTS
================================================================================

Event categories on the WebSocket stream:
- Telemetry events — raw and parsed live state changes.
- State updates — typed StateManager mutations.
- Critical alerts — overlay-eligible interrupting events (interdiction, hull
  critical, hostile contact); also routed to the overlay alert lane.
- Source events — registry health, queue/cache events, blocked or rate-limited
  responses, unsupported declarations.
- Workflow events — Operations/workflow engine lifecycle.
- Route events — Navigation lifecycle.
- Audit events — Activity Log mirror events the UI may show as toasts or in
  Why drawers.
- Overlay events — overlay banner lifecycle (glance and interruptive).
- Future squadron events — peer state, telemetry sync, role/authority,
  emergency/security, integrations (Phase 7+).

Payload principles
- Every event carries id, family, type, occurred_at, recorded_at, severity.
- Source-bearing events carry source, freshness, truth_class, caveat.
- AI-bearing events carry ai_draft_flag and grounding_refs.
- Gate-bearing events carry proposal_id and decision_state.
- Privacy-sensitive payloads are redacted; sensitive raw payloads stay in the
  Activity Log raw drawer (where allowed) and are not pushed over WebSocket.

23.1 Phase 1–3 (complete baseline)

  default.
- Activity Log: intel.*, route.*, source.*, exploration.*.

23.4 Phase 6 — Trade/Mining/Carrier/Colonization backend (Master §23, §32)

- Privacy: EDDN submission consent gating; CAPI auth gating; commander cargo

24. FUTURE-ROUTE BACKEND ACTIVATION
================================================================================

Squadrons (Phase 7)
- Pre-activation: Operations owns the Squadron package; group coordination is
  thin and limited to commander-entered notes plus future-flagged surfaces.
- Activation: Squadrons becomes a primary route with peer state, telemetry
  sync, invites, roles, shared operations, shared navigation, emergency/
  security, integrations. Operations → Squadron becomes a thin handoff.

Engineering (Phase 8)
- Pre-activation: Operations owns the Engineering package; planning is light;
  Intel owns raw materials and current loadout.
- Activation: Engineering becomes a primary route with goals, materials,
  blueprints, engineers/unlocks, Guardian/tech brokers, suit engineering,
  builds/loadouts, acquisition plan, sources/imports. Operations → Engineering
  becomes a thin handoff.

Carriers (reserved)
- Trigger: Fleet Carrier Full Command Center scope begins, or carrier-state
  load (ownership, tritium, market, storage, jump logistics, services,
  colonization support) overloads Operations → Carrier/Logistics.
- Pre-activation: Operations owns Carrier/Logistics objectives; Intel owns the
  owned-carrier summary and public carrier context; Navigation owns carrier
  routes; Settings owns CAPI consent for fleet carrier endpoints.
- Activation: Carriers becomes a primary route with persistent FC command
  state. Ownership transfers from Operations → Carrier/Logistics into Carriers,
  while Navigation and Intel retain their owned facts/routes and Carriers
  links to them.

Activation discipline
- Activation is a deliberate phase event, not an emergent feature. Each
  activation creates an explicit reconciliation entry in the Index and an
  explicit ownership-transfer note in the affected Operations package.

  presence test.

26. BACKEND GLOSSARY
================================================================================

Activity Log — canonical proof layer for backend events; UI affordances Why and
View Activity resolve to log entries.

ActiveRouteState — backend state object describing the current active route
sourced from NavRoute.json or an OmniCOVAS candidate.

AI Draft — TruthClass for AI-produced content that has not been promoted to a
recorded fact. AI Drafts are labeled.

AIProvider — backend interface that abstracts every LLM provider. NullProvider
must always work.

Backend Blueprint — this document; authoritative for backend service ownership.

Bookmark — saved location reference; owned by Navigation.

Broadcaster — async fan-out of typed events; isolates failing subscribers.

Bridge — FastAPI HTTP and WebSocket surface that feeds the Tauri UI.

CacheRecord — backend record of a previously fetched external response; carries
provider, endpoint, recorded-at, ttl, source confidence, and stale behavior.

Commander — the user; backend records actions taken on behalf of the commander
and never on the commander's behalf without consent.

Confirmation Gate — middleware that turns AI/voice/slash/route/external/
settings actions into commander-confirmed events.

Dashboard backend — backend assembly of compact projections feeding the
Dashboard UI.

Dispatcher — single typed event router from parsed events to handlers.

DPAPI vault — Windows-DPAPI-encrypted secret vault bound to the commander's
Windows account.

EntityRef — backend reference object for any in-game or in-OmniCOVAS entity.

Event family — top-level namespace of an event type (e.g., source.*, intel.*).

ExternalRequestBundle — executable per-provider request unit; owned by the
SourceRouter and queued by the ExternalRequestQueue.

ExternalRequestQueue — per-provider scheduler with bucket awareness and
backoff.

FieldProvenance — provenance object attached to every fact.

FreshnessState — TruthClass adjacent value indicating the freshness of a fact.

Intel backend — backend service group owning known information surfaces.

KnownFact — typed record of a single known fact with source, timestamp,
freshness, truth class, caveat, and Activity Log reference.

Local-first — design rule that local sources are checked before any external
provider.

Master Blueprint — constitutional source of truth for laws, principles,
pillars, phases, and high-level architecture.

NavRoute.json — Elite Dangerous companion file providing the in-game route;
treated as proof.

Navigation backend — backend service group owning routes and movement.

NullProvider — AIProvider implementation that performs no LLM calls.

Operations backend — backend service group owning session goals and workflow
execution.

Pin — Dashboard summary pin record; carries source, freshness, status, origin
route, Activity Log reference.

ProviderCapability — typed capability card per provider; mirrors Source
Capability Routing Reference Section 4.

Provenance registry — backend service that resolves field-level provenance for
the UI.

Resource Monitor — backend self-reporting of CPU, memory, latency, and async
queue depth.

RouteCandidate — typed proposed route; not yet active.

RouteIntent — typed commander intent for a route.

RouteJob — typed unit of work in the route source pipeline.

RouteSourceStatus — health record per route source.

SavedRoute — typed library entry.

Session Focus — Operations-owned active focus selector.

Source Capability Routing Reference — companion document; authoritative for
provider capability and budgets.

SourceRegistry — backend catalog of providers and their state.

SourceRouter — backend planner that turns a workflow into a WorkflowSourcePlan.

Spansh link-out — current Navigation pattern delegating route plotting to
Spansh in the browser.

StateManager — backend canonical state object.

Status.json — Elite Dangerous companion file with live ship/commander state.

Top Secret Mode — Squadrons reserved feature; gated by future security
doctrine.

Truth Class — categorical label of a fact's source posture; values defined in
Section 16.

UI Blueprint v1.0 — companion document; authoritative for frontend route
ownership.

WorkflowSourcePlan — backend plan-level contract for source-bound workflows;
required before any external request.
