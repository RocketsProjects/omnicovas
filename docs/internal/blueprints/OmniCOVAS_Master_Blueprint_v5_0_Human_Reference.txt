1. EXECUTIVE SUMMARY
================================================================================

OmniCOVAS is a local-first, source-backed, auditable command deck for Elite
Dangerous commanders. Its purpose is to consolidate live ship telemetry,
combat support, navigation, exploration, trading, mining, engineering,
Powerplay/BGS context, and squadron coordination into one coherent intelligence
suite while preserving commander sovereignty.

The project remains governed by the same constitutional direction established in
v4.x:
- AI suggests; the commander decides.
- Telemetry defines local reality.
- Unknown remains unknown.
- External services are respected, consent-gated, cached, batched, and bounded.
- Commander data stays local by default.
- Every meaningful action is auditable through the Activity Log.
- OmniCOVAS must work with no AI provider active.
- OmniCOVAS must not automate gameplay, bypass confirmation, or invent facts.

Master Blueprint v4.2 was intentionally broad. It contained the constitution,
architecture, roadmap, pillars, phase plans, source registry, voice assumptions,
backend assumptions, frontend assumptions, dependencies, and status dashboard in
one document. That was useful while the project was young, but it now creates
avoidable overlap and drift.

Master Blueprint v5.0 changes the document architecture. The Master Blueprint is
now the constitutional umbrella. It keeps laws, principles, project identity,
release philosophy, phase and pillar framework, authority rules, high-level
architecture, and doctrine. It delegates detailed UI, backend, provider, and
compliance decisions to companion authority files.

The purpose of v5.0 is therefore not to make the Master larger. The purpose is
to make the Master cleaner, more stable, and less likely to contradict the new
specialized authorities.

- source boundaries and unsupported claims;
- fallback wording;
- request budget posture;
- cache/batch/queue expectations;
- consent and authentication gates for source use;
- source relationship and contact guidance.

--------------------------
The Compliance Matrix is authoritative for legal and compliance truth. It owns:
- Frontier ToS posture;
- external service compliance;
- EDDN, EDSM, Inara, Spansh, EDAstro, EliteBGS, BGS-Tally compliance detail;
- external tool and dependency compliance;
- privacy law and commander privacy posture;
- API key handling compliance;
- AGPL, CLA, SBOM, attribution, and license obligations;
- compliance red flags and review schedule.

3.6 Index
---------
The Index is the lookup router. It tells a future reader where to look first.
It is not itself the final authority for the substance of a decision.

3.7 README, SECURITY, CONTRIBUTING, CLA, ADRs, accessibility docs, and phase guides
----------------------------------------------------------------------------------
These documents remain supporting authorities for their specific purposes:
- README: public repository face and user/developer quick summary.
- SECURITY.md: security reporting and security commitments.
- CONTRIBUTING.md: contribution process.
- CLA.md: contributor license agreement terms.
- ADR 0002: accepted Tauri plugin decision history.
- ADR 0003: accepted UI safe-rendering rule.
- NVDA smoke test: accessibility validation detail.
- Phase guides: execution detail for a phase; they do not override the Master,
  Compliance Matrix, UI Blueprint, Backend Blueprint, or Source Capability
  Routing Reference.

3.8 Conflict-resolution rules
-----------------------------
1. Compliance beats all implementation desire where law, ToS, privacy, license,
   attribution, or external-service rules are involved.
2. Master beats companion documents on laws, principles, pillars, phases, and
   constitutional doctrine.
3. UI Blueprint beats Master and Backend on route ownership and user-surface
   placement detail, except where a law or compliance rule forbids it.
4. Backend Blueprint beats lower-level docs on service ownership, state models,
   event models, workflow execution, API/bridge contracts, and backend
   implementation boundaries.
5. Source Capability Routing Reference beats Master/Backend/UI on provider
   capability, request budget details, unsupported facts, and fallback wording.
6. Index routes the lookup; it does not override the owning authority.
7. README and root docs are public-facing summaries; when they drift, update
   them after the authority docs are settled.
8. In real conflict, report the conflict, follow the highest authority, and add
   a reconciliation item rather than silently blending the documents.

4. CORE PHILOSOPHY
================================================================================

OmniCOVAS is:
- event-driven;
- async-first;
- modular by design;
- local-first for latency and privacy;
- source-backed for truth;
- auditable by default;
- privacy-protective by default;
- AI-provider agnostic;
- performance-first;
- commander-sovereign;
- zero-budget and community-sustainable;
- built for long-term maintainability.

The commander owns the system. OmniCOVAS must never become a black box that acts
without the commander, invents certainty, hides data flow, or turns external
services into unbounded dependencies.

Telemetry defines live reality. Sources define external knowledge. AI may help
classify, compare, explain, and draft plans, but AI is not a source of facts.

5. THE TEN LAWS OF OPERATION
================================================================================

Every module, document, route, service, workflow, and implementation decision is
subordinate to these laws. They are not suggestions.

Law 1 — Confirmation Gate
-------------------------
Mandate:
AI suggests. Commander decides. Always.

Requirements:
- The AI must never perform, trigger, or initiate an in-game action without
  explicit commander confirmation.
- Voice, slash command, route activation, workflow recommendation, external
  service write, or local setting change cannot bypass the Confirmation Gate
  when the action is protected.
- Commander confirmation must be scoped, auditable, and reversible where
  possible.
- No setting, macro, script, voice adapter, or plugin may disable the gate.

Delegated details:
- Backend gate contracts: Backend Blueprint.
- UI prompt/confirmation surfaces: UI Blueprint.
- Compliance boundaries for game actions: Compliance Matrix.

Hard stop:
Any design that allows unattended action, direct AI in-game action, or hidden
action execution is forbidden.

Law 2 — Legal Compliance Protocol
---------------------------------
Mandate:
Every applicable law, license, EULA, ToS, copyright, trademark rule, privacy
obligation, and provider policy is an absolute constraint.

Requirements:
- When uncertain, stop and verify.
- Do not treat “technically possible” as “permitted.”
- Do not bundle, scrape, automate, redistribute, or transmit in ways that the
  Compliance Matrix has not approved.

Delegated details:
- Compliance Matrix owns the legal details.
- SECURITY.md owns vulnerability reporting.
- CONTRIBUTING.md and CLA.md own contribution process and rights grants.

Hard stop:
Any proposal that conflicts with legal, license, ToS, privacy, or external
service terms must be redesigned, blocked, or deferred.

Law 3 — API Respect Protocol
----------------------------
Mandate:
External services are relationships, not infinite utilities. Rate limits,
provider terms, and maintainer goodwill are hard constraints.

Requirements:
- Cache and batch before calling.
- Use respectful request budgets, not provider hard limits as targets.
- Back off on rate-limit or health signals.
- Keep external calls visible in the Activity Log.
- Do not design one-call-per-field, one-call-per-card, or one-call-per-AI-line
  workflows.

Delegated details:
- Provider capability and budget posture: Source Capability Routing Reference.
- Queue/cache/source execution: Backend Blueprint.
- Provider compliance: Compliance Matrix.

Hard stop:
A workflow that requires uncontrolled external calls cannot ship.

Law 4 — AI Provider Agnosticism
-------------------------------
Mandate:
Any AI provider, or no AI provider, must be a commander choice. Core
OmniCOVAS functionality must work under NullProvider/no-AI mode.

Requirements:
- AI providers must sit behind an abstraction boundary.
- NullProvider must remain a first-class supported mode.
- Core telemetry, UI, Activity Log, settings, source routing, and deterministic
  workflows must not require an LLM.
- AI cannot be wired directly into backend services as a hidden dependency.

Delegated details:
- AI backend abstraction: Backend Blueprint.
- AI settings and UI behavior: UI Blueprint.

Hard stop:
A feature that makes a cloud AI provider mandatory for core operation violates
this law.

Law 5 — Zero Hallucination Doctrine
-----------------------------------
Mandate:
If it is not verified, it is not said as fact. Unknown remains unknown.

Requirements:
- Every displayed fact must have a source posture.
- Every recommendation must expose grounding, caveat, or confidence where
  relevant.
- AI output must be labeled as draft, synthesis, comparison, explanation, or
  recommendation, not source truth.
- Unsupported facts must use approved fallback wording.
- Mining hotspot/ring candidate claims, guaranteed market claims, known-ganker
  claims, and exact inventory claims require verified source support.

Delegated details:
- Fact provenance and truth class: Backend Blueprint.
- Provider support and fallback wording: Source Capability Routing Reference.
- UI source chips and wording surfaces: UI Blueprint.

Hard stop:
Do not invent Elite Dangerous mechanics, provider capabilities, market values,
route quality, source freshness, or commander data.

Law 6 — Performance Priority
----------------------------
Mandate:
Zero lag. Function before flair. Always.

Requirements:
- No blocking I/O in the core event loop.
- UI must stay responsive and must not steal focus during play.
- External calls must be queued, cached, and kept off hot paths.
- Resource budgets and latency budgets are constraints, not decorations.
- Visual effects must never outrank information clarity, accessibility, or game
  performance.

Delegated details:
- Backend resource monitoring and queue behavior: Backend Blueprint.
- UI density and overlay behavior: UI Blueprint.
- Phase-specific gates: active phase guide or future Development Roadmap.

Hard stop:
Any design that risks game stutter, hot-path blocking, or unbounded resource use
must be redesigned.

Law 7 — Telemetry Rigidity
--------------------------
Mandate:
Telemetry defines live local reality. External sources provide external
knowledge; they do not override local truth.

Requirements:
- Local Journal, Status.json, companion JSON files, and StateManager/cache are
  the primary truth path for the commander’s current session.
- Missing local fields stay missing unless a verified source supplies a labeled
  snapshot or context value.
- Inferred values must be labeled as inferred and must never override telemetry.
- External sources are not interchangeable; they are capability-bounded.

Delegated details:
- Source priority, state ownership, and event handling: Backend Blueprint.
- Provider capabilities and call-type boundaries: Source Capability Routing
  Reference.

Hard stop:
Do not use external APIs to fabricate live hull, shields, heat, fuel, pips, or
other local telemetry values.

Law 8 — Sovereignty and Transparency
------------------------------------
Mandate:
Commander data stays local by default. The system is always auditable.

Requirements:
- Outbound data is opt-in.
- Secrets are encrypted at rest and redacted from logs.
- No telemetry or analytics are sent to maintainers by default.
- Every meaningful action, source lookup, gate decision, export/delete,
  settings change, and external request is visible through the Activity Log.

Delegated details:
- Backend privacy enforcement and Activity Log records: Backend Blueprint.
- UI privacy/source controls: UI Blueprint.
- Legal/privacy detail: Compliance Matrix.
- Security reporting: SECURITY.md.

Hard stop:
Silent outbound data flow, secret leakage, or unlogged meaningful action is
forbidden.

Law 9 — Original Integration
----------------------------
Mandate:
OmniCOVAS builds its own core. It integrates compliantly, natively where
possible, and without making optional external tools required.

v5.0 update:
- Native-first voice is the current direction.
- EDDI is cut from current UI scope.
- VoiceAttack is optional adapter only.
- VoiceAttack is never required, never bundled, never the brain, and never a
  source of truth.
- OmniCOVAS is the brain.
- Future Commander-Confirmed Input Assist is compliance-gated only.

Requirements:
- Optional tools may support user convenience but cannot be critical to core
  operation.
- Integrations must preserve Law 1, Law 2, Law 5, Law 7, and Law 8.

Delegated details:
- Voice/Input UI posture: UI Blueprint.
- Voice/Input backend boundaries: Backend Blueprint.
- External tool compliance: Compliance Matrix.

Hard stop:
Do not reintroduce EDDI as a required dependency, make VoiceAttack required, or
allow any adapter to bypass confirmation.

Law 10 — Unified Intelligence, Independent Operation
----------------------------------------------------
Mandate:
OmniCOVAS is one command deck with independent core operation. Fragmentation and
critical external dependencies are failure states.

Requirements:
- Core functionality must operate with only OmniCOVAS and local telemetry.
- Optional external sources enhance knowledge but must degrade gracefully.
- Optional AI enhances synthesis but must not be required for core operation.
- Optional voice/input adapters enhance convenience but must not be required.
- Route, service, source, and workflow boundaries must prevent duplicated
  sources of truth.

Delegated details:
- UI route containment: UI Blueprint.
- Backend ownership and anti-duplication rules: Backend Blueprint.
- Source unsupported/fallback behavior: Source Capability Routing Reference.

Hard stop:
Do not create parallel StateManagers, dashboards-as-workflow-boards, operations
as fact encyclopedias, or external-tool dependencies for core functionality.

6. THE TEN ARCHITECTURAL PRINCIPLES
================================================================================

Principle 1 — Accessibility
---------------------------
OmniCOVAS must be usable by commanders with diverse abilities, input devices,
visual needs, and screen-reader workflows. Accessibility is a design
requirement, not a polish task.

Delegated details:
- UI implementation and route design: UI Blueprint.
- Accessibility smoke-test detail: NVDA smoke-test document.

Principle 2 — Sustainability
----------------------------
The project must remain maintainable by a zero-budget volunteer project. Avoid
unnecessary dependencies, recurring costs, cloud lock-in, and overbuilt systems.

Principle 3 — Evolutionary Design
---------------------------------
The project must adapt to Elite Dangerous updates, community feedback, provider
changes, and source availability without rewriting the core.

Principle 4 — Attribution
-------------------------
Every data source, dependency, contributor, external tool, and licensed asset
must be credited appropriately. Original OmniCOVAS UI art must not copy Frontier
assets.

Delegated details:
- Attribution/legal detail: Compliance Matrix.
- About route presentation: UI Blueprint.

Principle 5 — Graceful Failure
------------------------------
Every external dependency must fail safely. A missing source, disabled provider,
stale cache, failed AI provider, or unavailable adapter must produce clear
fallback wording and keep the app stable.

Principle 6 — Knowledge Base Stewardship
----------------------------------------
The Knowledge Base is project infrastructure. Entries must be source-backed,
versioned, reviewable, and compatible with the zero-hallucination doctrine.

Delegated details:
- Backend KB loader and validation: Backend Blueprint.
- Domain values and mechanics must not be invented by AI or executors.

Principle 7 — Privacy-by-Default
--------------------------------
The default posture is the most privacy-protective posture. Outbound data,
external authentication, provider lookups, sync, analytics, and sharing are off
unless explicitly enabled.

Principle 8 — Defensive Update
------------------------------
Game updates, dependency updates, schema changes, provider changes, and source
changes must be validated before being trusted. Assumptions must be testable and
failure must be visible.

Principle 9 — Explainability
----------------------------
Recommendations must be explainable. The commander must be able to see why a
claim or recommendation exists, what source supports it, and what caveat applies.

Principle 10 — Resource Efficiency
----------------------------------
CPU, memory, disk, network, queue depth, cache size, and UI rendering overhead
must be bounded and visible. OmniCOVAS must support play, not compete with it.

7. PROJECT ARCHITECTURE SUMMARY
================================================================================

7.1 High-level layers
---------------------
OmniCOVAS is organized as:

1. UI layer
   - Tauri desktop shell, overlay, route surfaces, settings, Activity Log, and

   - SourceRegistry, SourceRouter, WorkflowSourcePlan, ExternalRequestBundle,
     ExternalRequestQueue, cache, provenance, fallback wording.
   - Detailed authority: Backend Blueprint v1.0 plus Source Capability Routing
     Reference v1.

5. Compliance envelope
   - Legal, ToS, privacy, attribution, external service terms, licensing, CLA,
     SBOM, responsible disclosure.
   - Detailed authority: Compliance Matrix v4.1 and root support docs.

7.2 Activity Log as proof layer
-------------------------------
The Activity Log is the proof layer for OmniCOVAS. It records meaningful state
changes, source lookups, external requests, gate decisions, blocked requests,
settings changes, privacy actions, exports/deletes, AI drafts, and diagnostic
warnings. It does not own every route’s workflow, but every route must be able
to link to proof.

7.3 Confirmation Gate as action boundary
----------------------------------------
The Confirmation Gate is the action boundary. AI, voice, slash commands, route
activation, workflow recommendations, and external write-style operations cannot
bypass it.

7.4 No detailed tables here
---------------------------
The Master must not duplicate service registries, endpoint families, state
models, event taxonomies, route layouts, provider tables, or rate-budget tables.
Those belong in the companion documents.

8. DATA AND SOURCE PHILOSOPHY
================================================================================

8.1 Local-first truth
---------------------
Local Journal, Status.json, local companion JSON files, and OmniCOVAS state/cache
are checked before external providers. Local telemetry defines the commander’s
current session reality.

8.2 Field-level provenance
--------------------------
Every displayed fact must be capable of carrying:
- source;
- timestamp;
- freshness;
- truth class;
- caveat;
- Activity Log reference where meaningful.

8.3 Truth classes
-----------------
At a high level, OmniCOVAS distinguishes:
- live local telemetry;
- local snapshot;
- cached external data;
- authenticated snapshot;
- community-known data;
- commander-entered notes;
- inferred values;
- AI drafts;
- unknown / not loaded / disabled / stale / requires authorization / unsupported
  / no verified source.

Detailed truth-class values belong to the Backend Blueprint and Source
Capability Routing Reference.

8.4 Unknown remains unknown
---------------------------
If no verified source supports a fact, OmniCOVAS must not guess. The UI must
say Unknown, No Verified Source, Unsupported, Not Loaded, Disabled, Stale, or
Requires Authorization as appropriate.

8.5 Cache, batch, and respectful budgets
----------------------------------------
External calls must be cached, batched, queued, budgeted, and consent-gated.
Provider hard limits are not usage targets. Actual provider capability and
budget details belong to the Source Capability Routing Reference.

8.6 Source capability boundaries
--------------------------------
External sources are not interchangeable. EDSM, Ardent, CAPI, EDAstro,
EliteBGS, Spansh, Inara, EDDN, EDSY/Coriolis, and local files each have specific
roles. The Master does not redefine those roles; it delegates them to the Source
Capability Routing Reference.


================================================================================
9. AI DOCTRINE
================================================================================

9.1 AI provider agnosticism
---------------------------
OmniCOVAS may support Gemini, paid cloud providers, local LLMs, and NullProvider,
but no provider may become a core dependency. Provider implementation detail
belongs to the Backend Blueprint. Provider settings and user controls belong to
the UI Blueprint.

9.2 NullProvider and AI-off operation
-------------------------------------
NullProvider/no-AI mode must preserve core functionality. Telemetry, state,
settings, Activity Log, deterministic calculations, route surfaces, and source
routing cannot collapse when AI is disabled.

9.3 AI is not a source of facts
-------------------------------
AI may classify intent, summarize source-backed facts, compare options, draft a
WorkflowSourcePlan, write user-facing prose, or explain why a recommendation
exists. AI must not invent telemetry, prices, station inventory, BGS state,
route quality, hotspot quality, provider capability, commander data, or API
facts.

9.4 AI cannot call APIs directly
--------------------------------
AI may draft a request plan, but source routing and backend services decide what
facts can be requested. ExternalRequestBundles are queued by backend services,
not by the LLM.

9.5 AI cannot bypass the Confirmation Gate
------------------------------------------
AI output that proposes meaningful action must pass through the Confirmation
Gate before execution. This includes voice-adapter proposals, route activation,
external write-style actions, and commander-affecting local changes.

9.6 Grounding and explainability
--------------------------------
AI output must be grounded in telemetry, Knowledge Base entries, source-backed
facts, or commander-entered context. Where the AI synthesizes or recommends, the
output must remain explainable through source links, Why affordances, or
Activity Log entries.


================================================================================
10. VOICE AND INPUT DOCTRINE
================================================================================

10.1 Current direction
----------------------
The current voice direction is native-first. OmniCOVAS is the brain. Voice and
input systems are user interfaces or adapters, not independent decision-makers.

10.2 EDDI
---------
EDDI is cut from current UI scope. Master v4.2 wording that treats EDDI as a
required voice dependency is superseded by this v5.0 doctrine after adoption.
EDDI may remain historical context or future optional interop only if later
compliance and architecture review explicitly reintroduce it.

10.3 VoiceAttack
----------------
VoiceAttack is optional adapter only. It is never required, never bundled, never
the brain, never a source of truth, and never a Confirmation Gate bypass.

10.4 Commander-Confirmed Input Assist
-------------------------------------
Future input assist is compliance-gated and must remain:
- commander-requested;
- scoped to a short action;
- confirmed by the commander;
- auditable;
- safe;
- non-botting;
- non-unattended;
- compliant with Frontier and external-tool boundaries.

10.5 No unattended automation
-----------------------------
OmniCOVAS must not provide autopilot, auto-combat, auto-trading, auto-mining,
background route execution, anti-cheat bypass, process injection, or any direct
AI in-game action.


================================================================================
11. PRIVACY, SECURITY, AND COMPLIANCE DOCTRINE
================================================================================

11.1 Local by default
---------------------
Commander data, journals, settings, source caches, logs, and local state remain
on the commander’s machine unless the commander explicitly enables an outbound
flow.

11.2 Outbound opt-in
--------------------
Outbound data defaults off. CAPI, Inara, EDDN submission, commander profile
lookups, squadron sync, source provider authentication, and privacy-sensitive
external flows require explicit commander control.

11.3 Secret handling
--------------------
Secrets belong in the DPAPI-backed vault. API keys, OAuth tokens, credentials,
and raw sensitive config must not be logged. Redaction is mandatory before write.

11.4 No telemetry to maintainers
--------------------------------
OmniCOVAS must not send telemetry, analytics, crash data, or commander data to
project maintainers by default.

11.5 External flow auditability
-------------------------------
Every external request, blocked request, consent change, auth state change,
export/delete, and meaningful data-flow event must be visible through the
Activity Log or the Settings/Sources surfaces as appropriate.

11.6 Compliance Matrix ownership
--------------------------------
The Compliance Matrix owns legal, ToS, privacy law, license, attribution,
external service, and red-flag detail. The Master only states the doctrine.

11.7 SECURITY.md ownership
--------------------------
SECURITY.md owns vulnerability reporting, response timeline, recognition, and
security-reporting scope. The Master does not duplicate that process.

17. DEPENDENCY AND BLOCKER FRAMEWORK
================================================================================

The Master keeps dependency classes, not detailed task lists.

External approvals
------------------
Some source or integration workflows may require Frontier, Inara, or other
provider approval, whitelisting, or relationship management. Exact details belong
to the Compliance Matrix, Source Capability Routing Reference, and approval
application documents.

Source/provider relationships
-----------------------------
Provider hard limits are not usage targets. Provider communication, request
budget posture, and source capability verification must be maintained before
shipping workflows that depend on those sources.

Privacy/compliance gates
------------------------
CAPI, Inara, commander profile lookups, outbound submissions, sync, and
privacy-sensitive flows require explicit settings, auditability, auth handling,
and consent.

Code signing/trust
------------------
Release trust, signing, SBOM, dependency attribution, security posture, and
public distribution hardening remain release blockers as v1.0 approaches.

Rate/source capability verification
-----------------------------------
Provider capabilities, request shapes, and terms must be verified before use.
Candidate provider support does not equal an approved dependency.

Future voice/input assist compliance
------------------------------------
Commander-Confirmed Input Assist remains future and compliance-gated. No
unattended automation or botting is permitted.

Squadron security doctrine
--------------------------
Phase 7+ shared-state features require explicit privacy, security, encryption,
share-scope, emergency-action, and audit doctrine before implementation.

21. GLOSSARY
================================================================================

Activity Log
Canonical proof layer for meaningful events, source chains, external requests,
AI/gate records, confirmations, blocked requests, settings/privacy actions, and
diagnostics.

AI Draft
AI-produced text, comparison, plan, or recommendation that has not become a
source-backed fact.

AIProvider
Backend abstraction for LLM providers. NullProvider is the no-AI implementation.

Ardent
Candidate source for commodity buy/sell candidates and station service discovery
where supported by the Source Capability Routing Reference.

Backend Blueprint
Companion authority for services, state, events, workflows, API/bridge,
provenance, source execution, cache/queue, and backend privacy enforcement.

CAPI
Frontier Companion API. Authenticated, privacy-gated, not a public galaxy lookup.

Commander
The OmniCOVAS user. Commander sovereignty is a core doctrine.

Commander-Confirmed Input Assist
Future, compliance-gated input assistance where the commander requests, scopes,
confirms, and audits any protected action.

Confirmation Gate
Non-bypassable boundary between suggestion/proposal and protected action.

EDDN
Elite Dangerous Data Network. Strategic relay/future cache source, not a query
database.

EDSM
Source for system, station, body, context, and spatial candidates where supported;
not a broad global market search engine.

ExternalRequestBundle
Backend request unit generated after a WorkflowSourcePlan, queued and audited by
backend source execution.

Field-level provenance
Requirement that each displayed fact retain source, timestamp, freshness, truth
class, and caveat.

Index
Lookup router for where topics live. Not final authority by itself.

Intel
UI route/domain for what is known. Owns source-backed facts.

Local-first
Rule that local telemetry and local files are checked before external providers.

NullProvider
AIProvider implementation that performs no LLM calls and preserves core behavior.

Operations
UI route/domain for what the commander is doing. Owns active goals, workflows,
blockers, next actions, and debriefs; not raw facts or routes.

Source Capability Routing Reference
Companion authority for source/provider capability, request budgets, unsupported
facts, fallback wording, and source relationships.

Telemetry truth
Doctrine that local game telemetry defines live commander reality.

UI Blueprint
Companion authority for product, frontend route ownership, dashboard, overlay,
settings, About, route containment, and UI feature placement.

Unknown remains unknown
Rule that unsupported, missing, stale, unauthenticated, or unavailable facts are
labeled honestly instead of guessed.

VoiceAttack
Optional adapter only. Never required, never bundled, never the brain, never a
source of truth, and never a gate bypass.

WorkflowSourcePlan
Backend plan-level contract required before external request bundles are queued
or executed.
