Purpose
-------

This document is the authoritative OmniCOVAS source/provider routing reference.

Use it to decide:

1. Which source may answer a specific fact or workflow.
2. Which sources must not be used for that fact or workflow.
3. Whether consent, authentication, cache, queueing, rate budget, or fallback wording is required.
4. Whether a workflow is unsupported until a verified source exists.
5. What wording the UI must use when facts are missing, disabled, stale, unsupported, unverified, or unauthorized.

This document is intentionally source-bound. It does not invent provider capabilities, does not treat community tools as interchangeable, and does not let AI become a source of facts.

================================================================================
SECTION 1 — HIGHEST-PRIORITY ROUTING RULES
================================================================================

1. AI is not a source of facts.
   AI may classify commander intent and prepare a source request plan. AI must not invent telemetry, prices, station inventory, BGS state, route quality, hotspot quality, or API facts.

2. Local-first is mandatory.
   Local Journal, Status.json, local companion JSON files, and OmniCOVAS state/cache are checked before any external provider.

3. External sources are not interchangeable.
   A provider may only answer facts its documentation supports. A provider useful for one workflow must not be stretched into another workflow.

4. Unknown remains unknown.
   If no verified source supports a fact, the UI must say Unknown, Not Supported, No Verified Source, Not Loaded, Disabled, Stale, or Requires Authorization. Do not guess.

5. Field-level provenance is mandatory.
   Each displayed fact must retain source, timestamp, freshness, truth class, and caveat. Bundled requests are allowed, but bundled fields still need individual provenance.

6. Cache and batch before calling.
   Do not call one provider once per field, once per card, or once per AI sentence. Use cached facts and provider-supported batch endpoints wherever possible.

7. Respectful budgets are project policy.
   Provider-published hard limits are not usage targets. OmniCOVAS should use relaxed but respectful budgets, queue work, and back off immediately on provider rate-limit signals.

8. Consent and authentication are source gates.
   CAPI, Inara, commander profile lookups, outbound submissions, and any privacy-sensitive source require explicit commander control and Activity Log visibility.

9. No scraping by default.
   Websites without documented API permission are Link-Only or Format-Interop only. Do not design workflows around scraping.

10. Route activation and game actions require commander action.
    External route planning may produce a candidate. It must not automatically manipulate the game or bypass the Confirmation Gate.

SECTION 2 — EXACT CALL-TYPE ROUTER
================================================================================

Use this section first. It is the primary source selection table.

2.1 Live Commander / Ship State
-------------------------------
Use:
  1. Local Journal
  2. Status.json
  3. OmniCOVAS StateManager/cache
Fallback:
  - CAPI /profile only as an authenticated snapshot if local state is missing and CAPI is enabled.
Do not use:
  - EDSM, EDAstro, EDDN relay, Inara, Spansh, EliteBGS, Ardent.
UI wording:
  - “Live local telemetry” for local facts.
  - “Authenticated CAPI snapshot” for CAPI facts.

2.2 Hull, Shields, Pips, Fuel, Heat
-----------------------------------
Use:
  - Local Journal + Status.json + StateManager.
Fallback:
  - None for live truth.
Do not use:
  - External APIs for live values.
UI wording:
  - “Live local telemetry” or Unknown.

2.3 Current Cargo
-----------------
Use:
  1. Cargo.json
  2. Journal cargo events
  3. StateManager
Fallback:
  - CAPI snapshot if enabled and relevant.
Do not use:
  - EDDN relay, EDSM, Inara, Spansh, Ardent global data.
UI wording:
  - “Known local cargo.”

2.4 Current / Visited Station Market Snapshot
---------------------------------------------
Use:
  1. Local Market.json if Elite generated it.
  2. CAPI /market if enabled for last-docked station.
  3. EDSM or Ardent only as external community-known station/market context.
Do not use:
  - Inara API, EDDN relay directly.
UI wording:
  - “Local station snapshot” versus “community-known market data.”

2.5 Known Station Market by MarketID or Station Context
-------------------------------------------------------
Use:
  1. EDSM station market endpoint when marketId or exact station context is known.
  2. Ardent market/commodity endpoint for commodity-specific known-market facts.
Fallback:
  - Ardent system commodity endpoints.
Do not use:
  - Inara API, EDDN relay, EDAstro.
UI wording:
  - “Known station market data, freshness-labeled.”

2.6 Global Commodity Buy / Sell Candidates
------------------------------------------
Use:
  1. Ardent commodity imports/exports endpoints.
  2. Future OmniCOVAS EDDN-backed local cache.
Fallback:
  - Link-only/manual tools if no API source is enabled.
Do not use:
  - EDSM as a broad search engine.
  - Inara API.
  - CAPI.
  - EDDN relay directly.
  - EDAstro.
UI wording:
  - “Best known from enabled sources,” not “best in galaxy.”

2.7 Nearby Commodity Buy / Sell From Current System
---------------------------------------------------
Use:
  1. Ardent nearby imports/exports endpoints.
  2. Future OmniCOVAS EDDN-backed local cache.
Do not use:
  - Inara API, EDSM as direct broad search, EDDN relay directly.
UI wording:
  - “Nearby observed market candidates.”

2.8 Station Service Nearest Lookup
----------------------------------
Examples: refuel, repair, shipyard, outfitting, material trader, technology broker, black market, universal cartographics.
Use:
  1. Ardent nearest service endpoint.
  2. EDSM station context only when candidate systems/stations are already known and service facts are sufficient.
Important limitation:
  - Service availability is not exact module or ship inventory.
Do not use:
  - Inara API, EDAstro, EDDN relay directly.
UI wording:
  - “Nearest known station with service X.”

2.9 Exact Module Availability / Outfitting Inventory
----------------------------------------------------
Use:
  1. Local Outfitting.json for the current/visited station.
  2. CAPI /shipyard for last-docked station if enabled.
  3. EDSM known-station outfitting if exact station context is known.
Fallback:
  - Link-only/manual tools until a verified broad module inventory API exists.
Do not use:
  - Ardent nearest outfitting as exact module inventory.
  - Inara broad lookup.
UI wording:
  - “Known outfitting at selected station,” not “nearest guaranteed module.”

2.10 Shipyard / Ship Availability
---------------------------------
Use:
  1. Local Shipyard.json.
  2. CAPI /shipyard if enabled.
  3. EDSM known-station shipyard if exact station context is known.
Fallback:
  - Ardent nearest shipyard only for service presence.
Do not use:
  - Inara broad lookup unless future docs prove support.
UI wording:
  - “Known shipyard data.”

2.11 Public System Information
------------------------------
Examples: coordinates, permit, government, allegiance, economy, population, controlling faction, security, primary star.
Use:
  1. EDSM system endpoints.
  2. Ardent system lookup as fallback.
  3. Spansh dumps/local index if implemented and verified.
Do not use:
  - Inara API, CAPI for public lookup, EDDN relay directly.
UI wording:
  - “Known public system data.”

2.12 Nearby Systems / Spatial Candidate Systems
-----------------------------------------------
Use:
  1. EDSM sphere or cube endpoints.
  2. Ardent nearby systems.
  3. Spansh dumps/local index if implemented.
Known request-shape limits:
  - EDSM sphere radius: max 100 ly.
  - EDSM cube size: max 200 ly.
  - Ardent nearby maxDistance: max 500 ly; returns up to 1000 systems.
UI wording:
  - “Nearby known systems.”

2.13 System Bodies / Exploration Scan Context
---------------------------------------------
Use:
  1. Local Journal scan events for commander-observed facts.
  2. EDSM bodies / estimated value.
  3. EDAstro starsystem endpoint.
Fallback:
  - Spansh dumps/local index if verified.
Do not use:
  - Ardent trade endpoints, Inara API, EliteBGS.
UI wording:
  - “Commander-observed scan” for local data.
  - “Known body data” for external data.

2.14 Exploration POI / Galactic Exploration Catalog
---------------------------------------------------
Use:
  1. EDAstro GEC endpoints.
Fallback:
  - EDSM/Spansh POI or dump source only after verification.
Do not use:
  - Ardent, CAPI, Inara API, EDDN relay.
UI wording:
  - “Community POI.”

2.15 Route Planning / Neutron Route / Road to Riches
----------------------------------------------------
Use:
  1. Local NavRoute.json for current in-game route proof.
  2. Spansh link-out for immediate integration.
  3. Spansh API only after endpoint/rate/terms confirmation.
  4. Future local Spansh dump/router if implemented.
Support sources:
  - EDSM, Ardent, and EDAstro may enrich route stops; they are not route engines unless OmniCOVAS builds routing logic.
Do not use:
  - Inara API, EDDN relay, CAPI.

2.16 BGS Faction / Influence / State / History / Tick
-----------------------------------------------------
Use:
  1. EliteBGS.
  2. EDSM factions for secondary known-system context.
  3. Local Journal for commander-observed mission/faction effects.

Do not use:
  - Inara API for minor faction influences.

  - “Known BGS data from source X,” not official or final truth.

---------------------------------
Use:

AI role:

Do not use:

--------------------

Do not use:

2.19 Commander Profile / Ranks / Squadron
-----------------------------------------
Use:
  1. Local StateManager for in-session identity.

-----------------------------

Fallback:

Do not use:

2.21 Mining Hotspot / Ring Candidate
------------------------------------
Use:

Candidate source facts:

Fallback:
  - EDSY/Coriolis links for build planning, not inventory truth.
Do not use:
  - Inara API unless exact documented endpoint supports the fact.
UI wording:
  - “Known local material inventory” or “manual build link.”

2.23 Ship Build Planning
------------------------
Use:
  - EDSY/Coriolis link/export/import formats after format verification.
Source of truth:
  - Local Loadout/ModulesInfo for actual ship.
  - EDSY/Coriolis for planned build only.
Do not use:
  - Build tools as telemetry.
UI wording:
  - “Planned build” versus “current loadout.”

2.24 Traffic / Death Risk in a System
-------------------------------------
Use:
  - EDSM traffic/deaths.
Fallback:
  - Local commander history/cache.
Do not use:
  - Ardent trade volume as risk proxy unless clearly labeled as inferred.
UI wording:
  - “EDSM-reported traffic/deaths.”

SECTION 3 — PROVIDER PRIORITY MAP
================================================================================

P0 — Primary Local Truth
------------------------
1. Local Journal
2. Local companion JSON files
3. OmniCOVAS state/cache

Primary role:
  - Live commander reality, session telemetry, local snapshots, persisted source-backed state.

P1 — Highest-Utility External Query Sources
-------------------------------------------
1. Ardent
   - Best current candidate for EDDN-backed trade/market candidate search and station service lookup.
2. EDSM
   - Best current source for known-system, nearby-system, known-station, station market/shipyard/outfitting context.
3. Frontier CAPI
   - Best source for authenticated commander profile, last-docked market/shipyard, CG contribution, fleet carrier, visited stars.

P2 — Specialized Domain Sources
-------------------------------
1. EDAstro
   - Exploration, cartographic, known system contents, POI/GEC.
2. EliteBGS
   - BGS/faction/tick workflow candidate.
3. Spansh
   - Route planning link-out now; API/data-dump candidate after terms/rate verification.

P3 — Limited / Relay / Relationship-Dependent Sources
-----------------------------------------------------
1. Inara API
   - Useful only for supported Inara events, commander profile, and community goals. Not a universal lookup API.
2. EDDN
   - Strategic relay for future local cache and optional outbound submission. Not directly queryable.

P4 — Optional Interop / Link-Only / Retired
-------------------------------------------
1. EDMC, EDDiscovery, EDDI
   - Optional interop/reference; not required for core operation.
2. EDSY, Coriolis, Roguey, mEDI, EDTools, Miners Tool-style sites
   - Link-only or format-interop until API/terms are verified.
3. EDDB
   - Retired; do not plan active dependency.

SECTION 4 — PROVIDER CAPABILITY CARDS

4.1 Local Journal
-----------------
Source type:
  PRIMARY_LOCAL local file source.

Use for:
  - Current commander session events.
  - FSDJump, Docked, Undocked, LoadGame, Loadout, mission events, cargo events, combat events where exposed.
  - Activity Log source-chain roots.

Cannot provide:
  - Global market/outfitting/shipyard search.
  - Unobserved facts.
  - Complete mission board state.
  - Continuous values for every in-game UI state.

Request design:
  - File watch plus catch-up parse.
  - No external budget.
  - Performance bounded by local resource budget.

Implementation posture:
  - ACTIVE / CORE.

4.2 Local Companion JSON Files
------------------------------
Source type:
  PRIMARY_LOCAL local snapshot files.

Files:
  - Status.json
  - Cargo.json
  - Market.json
  - Outfitting.json
  - Shipyard.json
  - ModulesInfo.json
  - NavRoute.json

Use for:
  - Live ship status where reliable.
  - Local current/visited station market, outfitting, shipyard snapshots.
  - Cargo inventory.
  - Current in-game route if NavRoute.json exists.

Cannot provide:
  - Galaxy-wide lookup.
  - Facts before Elite writes the file.
  - Proof that missing file means the thing does not exist.

Request design:
  - File watch or safe polling.
  - Missing file = Not Loaded.

Implementation posture:
  - ACTIVE / CORE.

4.3 OmniCOVAS State / Cache
---------------------------
Source type:
  PRIMARY_LOCAL persisted state and cache.

Use for:
  - Current state assembled from source-backed facts.
  - Cached external responses.
  - Commander goals/notes.
  - Activity Log and provenance.

Cannot provide:
  - Live truth after data becomes stale.
  - Missing facts.
  - Higher authority than source telemetry.

Request design:
  - Every cached fact must retain provider, endpoint family, timestamp, TTL/freshness, source confidence, and stale behavior.

Implementation posture:
  - ACTIVE / CORE.

4.4 Ardent
----------
Source type:
  Public anonymous EDDN-backed REST API and data-dump/self-host candidate.

Best use:
  - Global and nearby commodity candidate searches.
  - Station service discovery.
  - EDDN-backed observed market data.

Verified can provide:
  - Commodity summaries.
  - Commodity importers/exporters globally.
  - Nearby imports/exports from a known system.
  - System lookup.
  - Nearby systems.
  - Nearest station by service.
  - System commodity data.
  - Market commodity data by marketId.
  - Database dumps / self-host option.

Cannot provide:
  - Official Frontier truth.
  - Private commander profile.
  - Live ship telemetry.
  - Route planning.
  - BGS influence/history unless a specific endpoint proves support.
  - Exact module inventory unless item-level outfitting endpoints are verified.
  - Guaranteed complete market coverage.

Request design:
  - Use maxDaysAgo, minVolume, minPrice/maxPrice, fleetCarriers intentionally.
  - Prefer nearby endpoints when origin is known.
  - Use local self-host/dumps for heavy workflows.

Operating budget:
  - Provider enforced limit: none currently identified.
  - Provider guidance: respectful use.
  - OmniCOVAS default: 6 requests/minute.
  - OmniCOVAS hard cap: 240/hour.

Contact route:
  - GitHub repository / maintainer issue route.

Implementation posture:
  - CANDIDATE / HIGH VALUE for Phase 6 trade and mining sell-side workflows.

4.5 EDSM
--------
Source type:
  Public API; authenticated commander endpoints for commander-specific features.

Best use:
  - Known system/station context.
  - Nearby systems.
  - Known station market/shipyard/outfitting checks.
  - Public exploration/body context.
  - Traffic/deaths and secondary BGS context.

Verified can provide:
  - System lookup by name.
  - Multi-system lookup.
  - Sphere systems, max radius 100 ly.
  - Cube systems, max size 200 ly.
  - Bodies and estimated values.
  - Stations.
  - Known station market, shipyard, outfitting.
  - Factions, traffic, deaths.
  - Authenticated commander logs endpoint with documented 360/hour limit for that endpoint only.

Cannot provide:
  - Live commander telemetry.
  - Global best market search.
  - Broad nearest exact-module search.
  - Route plotting.
  - Guaranteed current prices.
  - Complete mining hotspot workflow.
  - Official BGS strategy.

Request design:
  - Use batch/multi-system form where possible.
  - Use sphere/cube only for spatial workflows.
  - Use marketId when known.
  - Never call once per field.

Operating budget:
  - OmniCOVAS default public bucket: 2 requests/minute.
  - OmniCOVAS public hard cap: 120/hour.
  - Treat 429 as hard backoff.
  - Do not treat the logs endpoint 360/hour limit as universal permission for all EDSM endpoints.

Contact route:
  - EDSM footer routes: GitHub Issues, Discord, Frontier forum, social/contact channels.

Implementation posture:
  - CANDIDATE / HIGH VALUE for Phase 5+ system/station context.

4.6 Frontier Companion API / CAPI
---------------------------------
Source type:
  Authenticated official companion API; privacy-gated.

Best use:
  - Commander-authenticated profile and last-docked station data.

Verified can provide:
  - /profile commander snapshot.
  - /market last docked station/fleet carrier/settlement market.
  - /shipyard last docked shipyard and outfitting data.
  - /communitygoals active CGs and commander contributions.
  - /journal authenticated commander journal data, not real-time.
  - /fleetcarrier authenticated carrier owner data.
  - /visitedstars visited stars archive generation/download flow.

Cannot provide:
  - Public galaxy lookup.
  - Broad market search.
  - Route plotting.
  - Real-time journal stream.
  - Unauthenticated profile lookup.
  - Background polling without consent.

Request design:
  - Consent-gated.
  - Tokens stored in DPAPI vault and never logged.
  - Low frequency.
  - /journal is session-end/backfill, not live gameplay.

Operating budget:
  - Provider guidance: generally no more than 1 request/minute.
  - OmniCOVAS default: 1 request/minute.
  - OmniCOVAS daily default: 60/day until project relationship is established.

Contact route:
  - Frontier developer application / CAPI approval route.

Implementation posture:
  - FUTURE / PRIVACY-GATED / HIGH VALUE after approval.

SECTION 5 — UNSUPPORTED OR NOT-YET-VERIFIED FACTS
================================================================================

These facts must not be presented as supported until a verified source is added.

Unsupported facts/workflows:
  - Best mining hotspot near me.
  - Best ring candidate as a complete workflow.
  - Guaranteed best sell station in the galaxy.
  - Nearest station selling exact module X as a broad workflow.
  - Global mission board search.
  - Complete current mission list if OmniCOVAS did not observe/persist it.
  - All station inventories everywhere.
  - Official BGS strategy.
  - Real-time CAPI journal.
  - Query EDDN for latest market.
  - Inara market/faction/system/station API lookup.
  - Guaranteed all exobiology opportunities.
  - Automatic route activation in-game.
  - Website scraping as a backend.

Required UI wording:
  - Unsupported: “No verified source supports this yet.”
  - Disabled source: “Source disabled.”
  - Auth missing: “Requires commander authorization.”
  - Stale cache: “Cached from {source}, last updated {timestamp}.”
  - Partial coverage: “Partial result from enabled sources.”
  - Community-observed coverage: “Known/observed, not guaranteed complete.”

SECTION 7 — IMPLEMENTATION GUARDRAILS
================================================================================

1. Add source capability checks before every external request.
2. Make unsupported facts testable.
3. Add unit tests for disabled, stale, missing, unsupported, rate-limited, and auth-required states.
4. Log every external request and cache hit to the Activity Log.
5. Expose source/freshness in the UI.
6. Add Data Flow / Sources settings before enabling external workflows.
7. Treat maintainer feedback as higher authority than project defaults.
8. Prefer local indexes and dumps for heavy repeated workflows.
9. Separate “current local truth” from “community-known external data.”
10. Never promote a candidate provider to active dependency until terms, endpoint payloads, attribution, limits, and fallback behavior are verified.
