The most important rule is that features are not “placed” merely by naming a route. Every feature must have a true containment pattern, a primary surface, secondary surfaces, suppressed surfaces, link targets, and a non-duplication rule.

1. Executive Summary
--------------------
OmniCOVAS is a command deck, not a generic web dashboard. The final UI model is organized around precise route ownership:

- Dashboard = what matters right now.
- Intel = what is known.
- Navigation = where / how to move.
- Operations = what I am doing.
- Activity Log = how we know / what happened.
- Settings = how the app behaves.
- About = what the project is.
- Squadrons = group coordination, active Phase 7+.
- Engineering = progression planning, active Phase 8+.
- Carriers = fleet carrier command, reserved for FC Full Command Center.

The biggest v1.0 correction is that future features must not be assigned vaguely to Operations. Operations owns active intent, goals, checklists, blockers, and next actions. It must link outward to Intel for known facts and Navigation for routes/movement. This prevents Operations from becoming a cluttered duplicate of every route.

No primary route is created for Combat, Exploration, Exobiology, Trade, Mining, BGS, Powerplay, Passenger, Legal, Credits, SBOM, Polish, Security, Performance, KB, Explainability, or Release. These are contained through existing route families unless a future route-promotion test proves otherwise.

4. Core Route Philosophy
------------------------
The final UI model is based on route separation:

Dashboard
  Live command surface. It shows what matters right now.

Intel
  Known information console. It owns facts, context, personal ledgers, source-backed values, and known state.

Navigation
  Route and movement console. It owns plotting, active routes, route libraries, bookmarks, waypoints, route sources, and route candidates.

Operations
  Session operations console. It owns active intent, goals, workflows, progress, blockers, next actions, and debrief summaries.

Activity Log
  Data audit console. It owns proof, source chains, external request records, AI/gate records, confirmations, blocked requests, unknowns, and unsupported facts.

Settings
  Configuration console. It owns behavior, privacy, source setup, AI, Voice, overlay profile, diagnostics, import/export, and session-focus defaults.

About
  Compact project/reference route. It owns product identity, links, acknowledgments, license/security/reporting references, and document pointers.

5. Global Route Ownership Rule
------------------------------
A route may consume summaries from another route, but it must not duplicate that route’s full ownership surface.

Non-owning routes may:
- show compact linked summaries;
- show status chips;
- show blockers / unknowns;
- link to the owning route;
- pin a compact source-backed summary to Dashboard when commander-selected.

Non-owning routes must not:
- duplicate full fact views;
- duplicate full route views;
- duplicate full workflow workspaces;
- duplicate audit trails;
- duplicate settings controls;
- create parallel source-of-truth panels.

6. Feature Containment Doctrine
-------------------------------
A future feature is not represented just because it has a route home. Every feature must define:

1. Primary surface — where the commander actively uses it.
2. Secondary surfaces — where it may appear as a summary, chip, pin, alert, shortcut, or linked reference.
3. Suppressed surfaces — where it must not appear to avoid clutter or duplicate ownership.
4. Display pattern — card, row, drawer, tab, modal, timeline entry, wizard, pin, overlay banner, command result, or settings panel.
5. Information density — glance, summary, detail, or audit.
6. Alert behavior — silent record, passive status, glance notice, interruptive alert, or commander-confirmed action proposal.
7. Source/provenance behavior — inline chip, source drawer, Why link, Activity Log reference, or unsupported fallback.
8. Lifecycle — when it appears, collapses, archives, and becomes a notification.
9. User control — pin, mute, collapse, filter, move, request lookup, or view proof.
10. Non-duplication rule — what feature/route/source-of-truth it must not duplicate.

7. Route Expansion and Promotion Doctrine
-----------------------------------------
Do not create primary routes casually. Future features should first be assigned to Dashboard, Intel, Operations, Navigation, Activity Log, Settings, About, or Overlay behavior.

A future feature cluster may receive a reserved or active primary route only if it passes the Route Promotion Test:

1. Large enough — many commander-facing workflows/states would overload an existing route.
2. Persistent enough — ongoing commander context, not a temporary workflow/card/modal/source lookup.
3. Distinct enough — clear identity not merely a sub-area of an existing route.
4. Cross-cutting enough — coordinates operations, shared state, permissions, privacy, source health, audit trails, or user roles.
5. Commander-facing enough — benefits from a dedicated workspace.
6. Non-redundant — unique owner statement and no duplication of existing routes.
7. Auditable and bounded — preserves Activity Log ownership of proof and Settings ownership of configuration.

10. Dashboard Route Definition
------------------------------
Dashboard is the Live Command Surface. It provides the commander’s immediate ship/session picture, source-backed summaries, route/workflow/context shortcuts, and AI/command entry point.

Dashboard structure:
1. Fixed Top Core Internal Bar — Hull, Shields, Fuel, Heat, Pips.
2. Optional Top Internal Slots — compact internal ship/session values.
3. Left Internal Data Rail — five configurable internal data cards.
4. Center Ship Display — original OmniCOVAS ship visual and registry plate.
5. Right External Data Rail — five configurable external/contextual summary cards.
6. Fixed Bottom Core External Bar — Location, Status, Local, Objective, Data.
7. Dashboard Pins — source-backed compact summaries from owning routes.
8. Bottom AI / Command Bar — natural-language input, deterministic slash commands, source/audit shortcuts.

Dashboard owns:
- live safety glance;
- fixed top core ship telemetry;
- fixed bottom session/context posture;
- optional commander-selected internal summaries;
- optional commander-selected external summaries;
- source-backed pins;
- compact links into Intel, Navigation, Operations, Activity Log, Settings, and About;
- natural-language and slash-command entry.

Dashboard does not own:
- full known-information views;
- route planning/tracking/bookmarks;
- active workflow execution;
- audit/source history;
- settings/configuration;
- project/legal/reference documentation.

Dashboard must remain restrained. It must not become a freeform widget board, route planner, Intel encyclopedia, Operations board, Activity Log clone, or Settings panel.

Fixed Top Core Internal Bar
  Hull, Shields, Fuel, Heat, and Pips are always visible and non-customizable. They own live ship/session safety glance. They do not own detailed combat interpretation, long-term analytics, loadout planning, Engineering planning, or audit detail.

Optional Top Internal Slots
  Candidate values: Jump Range, Cargo, Modules, Hardpoints, Silent Running, Ship Computer, Repair/AFMU, SRV/Suit, Materials, Engineering State. They are summary portals, not mini-routes.

Left Internal Data Rail
  Internal summaries only: ship, cargo, modules, suit/SRV, utilities, repair, materials, internal state. Links cargo/material/loadout facts to Intel -> Ship / Personal. Links combat readiness to Operations -> Combat. Links engineering-related status to Engineering when active.

Center Ship Display
  Original OmniCOVAS wireframe/vectograph style only. No Frontier assets, traced ships, screenshots, or copied art. Visual overlays never replace text values.

Right External Data Rail
  External/contextual summaries only. Examples: Navigation, Operations, System Intel, Local Intel, Market/Trade, BGS, Squadron, External Sources, future route summaries. It does not duplicate full route/fact/workflow/source views.

Fixed Bottom Core External Bar
  Location links to Intel -> System/Local or Navigation -> Active Route if route-linked.
  Status links to Activity Log, Operations, or Intel as relevant.
  Local links to Intel -> Local.
  Objective links to Operations and route-linked Navigation.
  Data links to Settings -> Sources / Data Flow, Settings -> Privacy & Data, and Activity Log.

Dashboard Pins
  Pins require source, freshness, status, origin route, and Activity Log reference. Status values: Active, Stale, Unavailable, Disabled, Commander-entered, Coming Later. Pins must never pretend cached data is live.

Bottom AI / Command Bar
  Owns natural-language input, deterministic slash commands, command autocomplete, AI/provider shortcut, and source/audit shortcuts. It does not own facts, route execution, workflows without confirmation, direct API calls, or protected actions.

11. Intel Route Definition
--------------------------
Intel is the Known Information Console. Intel owns source-backed known information. It does not own active workflow execution, route planning, settings, or audit history.

Internal Intel locations:

1. Galaxy
  Galaxy-wide known information:
  - Community Goals;
  - galaxy POIs;
  - GalNet/community links where source-backed;
  - galaxy-scale exploration context;
  - broad BGS/Powerplay context;
  - community events/notices;
  - Powerplay-wide context;
  - BGS-wide/tick context if source-backed;
  - broad source-backed galactic opportunities.

2. System
  System-wide known information:
  - system identity, coordinates, permit, government, economy, security;
  - controlling faction, population, allegiance;
  - known bodies;
  - known stations/carriers/settlements;
  - system factions/BGS/Powerplay context;
  - traffic/death risk;
  - system-level exploration context;
  - system-level POIs;
  - route relevance as context only;
  - operation relevance as context only.

3. Local
  Specific local-location known information:
  - station/carrier/settlement facts;
  - services;
  - market/outfitting/shipyard known data;
  - body/ring/site facts;
  - landing context;
  - biological/geological/local signals;
  - settlement/surface/POI context;
  - current local activity context.

  Local context types:
  - Station / Carrier / Settlement;
  - Body / Planet / Moon;
  - Ring / Belt / Resource Site;
  - Surface Site / POI;
  - Current local activity context.

4. Ship / Personal
  Commander-owned and personal known information:
  - commander profile facts;
  - credit balance where source-backed;
  - ranks/reputation/status;
  - current ship facts;
  - cargo/loadout/material summaries;
  - known active missions;
  - unsold exploration data value;
  - unsold exobiology data value;
  - personal discovery/session ledger;
  - authenticated CAPI/profile facts where enabled;
  - current ship identity, ship type/name/identifier, and source-backed rebuy;
  - owned carrier summary until/unless Carriers becomes active route;
  - commander-entered facts where allowed.

All Intel facts must show source, timestamp/freshness, truth class, caveat, and Activity Log link where relevant.

Intel link targets:
- intel:galaxy, intel:galaxy/community-goals, intel:galaxy/poi, intel:galaxy/powerplay, intel:galaxy/bgs-tick, intel:galaxy/events.
- intel:system:<system_name>, /overview, /bodies, /stations, /factions, /bgs, /traffic-risk, /exploration.
- intel:local:<system>/<station|body|ring|settlement|site>, /services, /market, /outfitting, /shipyard, /landing, /signals, /bio, /rings.
- intel:personal, /commander, /ship, /cargo, /loadout, /materials, /ranks, /missions, /exploration-data, /exobiology-data, /credits, /fleet-carrier.

  - mission route sets;

13. Operations Route Definition
-------------------------------
Operations is the Session Operations Console. It owns active commander intent, Session Focus, goals, workflows, progress, blockers, linked route summaries, linked Intel summaries, next actions, and operation debriefs.

Operations does not own:
- known facts, which belong to Intel;
- route planning/tracking/bookmarks, which belong to Navigation;
- audit/source history, which belongs to Activity Log;
- configuration, which belongs to Settings;
- compact live command display, which belongs to Dashboard.

Internal Operations locations:
1. Command Board — active operation overview, blockers, linked route, linked Intel, next action.
2. Session Focus — live focus selector and impact preview.
3. Missions — known mission objectives, progress, blockers, linked destinations.
4. Combat — active combat workflows using the locked Combat package structure.
5. Exploration / Exobiology — active exploration, scanning, sampling, survey, redemption objectives.
6. Trade / Mining — active trade, hauling, mining, cargo, market, sell/buy objectives.
7. Passenger — active passenger mission objectives and destination workflow.
8. BGS / Powerplay — active campaign objectives; Intel owns facts, Operations owns campaign.
9. Carrier / Logistics — carrier, tritium, supply, expedition, colonization objectives until/unless Carriers activates.
10. Engineering — active engineering goals until Engineering route activates; after activation, thin handoff only.
11. Squadron — active group-operation goals until Squadrons route activates; after activation, thin handoff only.
12. Custom / Manual — commander-entered objectives and manual checklists.

Operations link targets include operations:board, operations:focus, operations:missions, operations:combat, operations:exploration, operations:trade-mining, operations:passenger, operations:bgs-powerplay, operations:carrier-logistics, operations:engineering, operations:squadron, operations:custom.

Operations must link outward for facts, routes, proof, settings, and pins. It must not recreate Intel panels, Navigation planners, route libraries, bookmark managers, source/audit tables, or settings controls.

14. Activity Log Route Definition
---------------------------------
Activity Log is the Data Audit Console. It answers what happened, how OmniCOVAS knows, what source produced it, what changed, what consumed the data, what was external, what was AI, what was confirmed, what was blocked, what was exported/deleted, and what was stale/unknown/unsupported.

Internal locations:
- Timeline;
- Audit Explorer;
- Source Chains;
- External Requests;
- AI / Gates;
- Exports;
- Errors / Diagnostics.

Views:
1. Commander-Relevant — default.
2. All Events — deep audit/debug.

Other routes should show source chips, Why affordances, View Activity links, freshness labels, and compact source drawers. Activity Log holds the deep detail.

17. Overlay Role Definition
---------------------------
Overlay is the in-game glance/interruption layer, not a second Dashboard.

Overlay owns:
- time-sensitive glance notices;
- critical/interruption alerts;
- compact command prompts when commander confirmation is required;
- in-game readable summaries driven by owned route state.

Overlay does not own:
- full Dashboard state;
- route details;
- full operations workspaces;
- Intel pages;
- Settings controls;
- Activity Log detail.

Alert lanes:
1. Silent Record — Activity Log only.
2. Operations Status — badge/summary.
3. Dashboard Optional Pin — commander-selected.
4. Overlay Glance Notice — short non-blocking notice.
5. Overlay Interruptive Alert — immediate danger or commander-confirmed action proposal only.

18. AI / Voice / Slash Command Doctrine
---------------------------------------
AI is not a source of facts. AI may classify intent, explain, summarize, compare, and prepare WorkflowSourcePlans. OmniCOVAS source routing resolves allowed facts and ExternalRequestBundles.

Native-first voice is the current UI direction. EDDI is cut from current UI scope and must not be reintroduced as an active dependency. VoiceAttack is optional adapter only, never required, never bundled, never a source of truth, and never a Confirmation Gate bypass.

Slash commands are deterministic app shortcuts, not workflow engines. Complex requests are AI workflows, not slash-command workflows.

Visible command help must use:
/open dashboard|intel|operations|navigation|activity|settings|about

The old /open legal form may remain as a legacy alias that opens About, but visible UI route naming must use About.

19. Source-Routing and Wording Doctrine
---------------------------------------
If older feature names imply stronger source support than OmniCOVAS has, UI Blueprint v1.0 renames or qualifies the feature.

Examples:
- “PvP Gank Database” becomes “PvP Encounter Log & Risk Intel.”
- “Best hotspot near me” is forbidden unless a complete verified source exists.
- “Trade Loop Optimizer” becomes “Trade Loop Candidate Review” or equivalent if sources support candidates, not certainty.
- “Known ganker” must not appear unless a verified source supports that exact claim.
- “Nearest guaranteed module” must not appear when only service availability or stale station outfitting data exists.

Preferred UI wording:
- Local telemetry;
- Known local state;
- Known active missions;
- Community-known data;
- Source-backed candidate;
- Commander-entered note;
- Requires Authorization;
- Disabled;
- Stale;
- Unsupported;
- No Verified Source.
