3. ROOT CAUSE SUMMARY
================================================================================

Two defects interact to produce the observed symptom:

Defect 1 — Schema confusion on the Loadout event.

   The field "FuelCapacity" has two different shapes depending on the event:

      LoadGame   : FuelCapacity is a scalar float (e.g. 128.0) and there is
                   a separate FuelLevel scalar (e.g. 128.0 at session start).
      Loadout    : FuelCapacity is an object {"Main": float, "Reserve": float}
                   describing tank geometry. There is NO FuelLevel field.

   The current Loadout handler appears to treat FuelCapacity uniformly as
   if it were a scalar carrying current fuel. The result is that state.ship
   receives the capacity value (128.0) and treats it as the live level.

Defect 2 — Missing live-gauge subscription.

   The Core Brain boot log emits:

      "Fuel handlers registered (FuelScoop, RefuelAll, RefuelPartial,
       StartJump)"

   ReservoirReplenished is not registered. This is the dominant live-gauge
   event during supercruise and normal flight. In the 2026-04-20 session it
   fired 45 times in 40 minutes. Without this handler, the only updates to
   fuel_level between station refuels are FuelScoop events, and those only
   fire during active scooping.

   StatusReader DOES read Status.json Fuel and DOES trigger FUEL_LOW and
   FUEL_CRITICAL broadcasts on threshold crossing, but it writes the reading
   into a transient comparison variable rather than persisting it to
   state.ship. The threshold logic works; the state hydration does not.

Result: state.ship.fuel_level (or its equivalent) is seeded from
Loadout.FuelCapacity at session start and is never corrected.

6. STATE CONTRACT — REQUIRED FIELDS ON state.ship
================================================================================

After this Playbook lands, state.ship must expose exactly these four fuel
fields. Types are concrete, Optional where the field has no ground-truth
until a source has fired at least once.

   fuel_main              : float | None    live main tank (tonnes)
   fuel_reservoir         : float | None    live reservoir (tonnes)
   fuel_capacity_main     : float | None    main tank geometry (tonnes)
   fuel_capacity_reserve  : float | None    reservoir geometry (tonnes)

Rules:

   A. Initial value on a fresh Ship instance: all four None.

   B. None means "not yet observed". Do not coerce to 0.0. Do not coerce to
      the capacity value. Law 5 (Zero Hallucination): unknown stays unknown
      until a real source fires.

   C. If state.ship currently carries a scalar field called fuel_level or
      fuel_capacity (confirmed in V1), resolve as follows:

          - If fuel_level exists and is used ONLY by fuel.py and the [STATE]
            print: remove it. Replace all reads with (fuel_main or 0.0) +
            (fuel_reservoir or 0.0) for display, and None-check for any
            logic that needs a live gauge.
          - If fuel_level exists and is read by code outside fuel.py, status
            print, or tests: STOP. Escalate per Section 15. Do not rename.

   D. The [STATE] periodic print in main.py may continue to display a single
      "fuel=" value. Compute it as (fuel_main or 0.0) + (fuel_reservoir or
      0.0). When both are None, print "fuel=?" instead of a number. This
      makes the zero-hallucination posture visible in logs.

   E. Comment each new field with a one-line reference to this Playbook
      (e.g. # PB-01 fuel live gauge) and to Blueprint Section 19.C.5.

7. HANDLER CONTRACTS (omnicovas/features/fuel.py)
================================================================================

This section defines, per event, exactly what the handler reads and exactly
what it writes to state. Handlers are async, type-annotated, and conform to
the existing Week 7 handler shape in ship_state.py and fuel.py.

A. ReservoirReplenished (NEW HANDLER — highest priority)

   Event schema:
      { "event":"ReservoirReplenished",
        "FuelMain": float,
        "FuelReservoir": float }

   Reads:   event["FuelMain"], event["FuelReservoir"]
   Writes:  state.ship.fuel_main       = FuelMain
            state.ship.fuel_reservoir  = FuelReservoir
   Broadcasts: call the existing threshold evaluator (Section 10) AFTER the
               write. Do not emit any new event type.

   Implementation note: this event fires every few seconds during supercruise.
   It is the dominant live-gauge signal. Handler must complete well under the
   combat-alert budget of 100 ms. Typical execution should be under 1 ms —
   two scalar writes plus a threshold check.

   event_types.py: add RESERVOIR_REPLENISHED = "ReservoirReplenished" if
   absent (confirmed in V5).

B. Loadout (CORRECT EXISTING HANDLER — in ship_state.py, not fuel.py)

   Event schema:
      { "event":"Loadout",
        "FuelCapacity": { "Main": float, "Reserve": float },
        ... }

   Reads:   event["FuelCapacity"]["Main"], event["FuelCapacity"]["Reserve"]
   Writes:  state.ship.fuel_capacity_main    = FuelCapacity.Main
            state.ship.fuel_capacity_reserve = FuelCapacity.Reserve
   DO NOT WRITE fuel_main or fuel_reservoir from Loadout. Loadout carries no
   live fuel level — this is the core of Defect 1.
   Broadcasts: none (existing LOADOUT_CHANGED broadcast stays as-is).

   Defensive parsing: if event["FuelCapacity"] is a float rather than a
   dict, log a structlog warning tagged schema.fuel_capacity.shape and treat
   it as {"Main": value, "Reserve": 0.0}. This is belt-and-braces against
   Frontier changing the schema back in a future patch.

C. LoadGame (CORRECT EXISTING HANDLER — in ship_state.py)

   Event schema:
      { "event":"LoadGame",
        "FuelLevel": float,
        "FuelCapacity": float,
        ... }

   Reads:   event["FuelLevel"], event["FuelCapacity"]
   Writes:  state.ship.fuel_main           = FuelLevel
            state.ship.fuel_capacity_main  = FuelCapacity
   LoadGame does not provide reservoir information. Do not touch
   fuel_reservoir or fuel_capacity_reserve here — they will be populated by
   the Loadout event that always follows LoadGame within seconds.
   Broadcasts: none.

D. FSDJump (NEW OR CORRECT EXISTING HANDLER — in fuel.py or ship_state.py)

   Event schema:
      { "event":"FSDJump",
        "FuelUsed": float,
        "FuelLevel": float,
        "StarSystem": str, ... }

   Reads:   event["FuelLevel"]
   Writes:  state.ship.fuel_main = FuelLevel
   Broadcasts: call the threshold evaluator after the write.

   Note: if ship_state.py already handles FSDJump for system-name updates,
   extend that handler with one additional line rather than registering a
   second FSDJump handler. Two handlers on the same event is allowed but
   adds dispatcher overhead; prefer in-place extension.

E. FuelScoop (CORRECT EXISTING HANDLER)

   Event schema:
      { "event":"FuelScoop",
        "Scooped": float,
        "Total": float }

   Reads:   event["Total"]
   Writes:  state.ship.fuel_main = Total
   Broadcasts: call the threshold evaluator after the write. FuelScoop
   crossing UP through a threshold must not re-broadcast FUEL_LOW or
   FUEL_CRITICAL — the threshold-crossing state machine already handles
   this correctly per existing Week 7 logic; verify it still does after
   your changes.

   Secondary cleanup: locate the "fuel dropped below 25%%" log string and
   fix the double-percent. This is the cosmetic defect in the [STATE]
   channel of the FUEL_LOW broadcast. Change "25%%" to "25%".

F. RefuelAll, RefuelPartial (CORRECT EXISTING HANDLERS)

   Event schema (both):
      { "event":"RefuelAll" | "RefuelPartial",
        "Cost": int,
        "Amount": float }

   These events do not carry a post-refuel fuel level. The correct behaviour
   is to increment state.ship.fuel_main by event["Amount"], capped at
   state.ship.fuel_capacity_main if that value is known.
   Broadcasts: threshold evaluator after write. RefuelAll will almost always
   cross the FUEL_LOW threshold upward; that must not re-broadcast.

G. StartJump (NO CHANGE)

   StartJump has no fuel payload. Existing handler, whatever it does with
   StartJump, is out of scope for this Playbook.

8. STATUS.JSON POLLING CONTRACT (omnicovas/core/status_reader.py)
================================================================================

StatusReader polls Status.json at 2 Hz (every 0.5 s per Week 7).

On each poll, if the parsed payload contains a "Fuel" key with a dict value,
apply the following writes:

   state.ship.fuel_main      = payload["Fuel"]["FuelMain"]
   state.ship.fuel_reservoir = payload["Fuel"]["FuelReservoir"]

Rules:

   A. Do this write unconditionally when the Fuel key is present. No
      threshold evaluation on the status_reader path; thresholds are
      evaluated only on journal events (Section 10 + Section 9).

   B. Absence of the Fuel key means the game is in a state where fuel is
      undefined (main menu, loading screens — Status.json Flags:0 in the
      uploaded sample confirms this). In that case, do not modify
      fuel_main or fuel_reservoir. Do not clear them to None. The last
      known good value persists until a new source overwrites it.

   C. No broadcast from status_reader. The journal handlers carry the
      broadcast responsibility per Section 9.

   D. Under no circumstances should status_reader write to
      fuel_capacity_main or fuel_capacity_reserve. Status.json does not
      carry capacity.


================================================================================
9. SOURCE PRIORITY (LAW 7 ENFORCEMENT)
================================================================================

Per Blueprint Section 2 Law 7 and phase_2_dev_guide.txt line 260-261:

   "Source priority: journal wins for discrete events, status_json wins
    for live gauge."

Concrete rule set:

   1. Journal events (ReservoirReplenished, FuelScoop, FSDJump, LoadGame,
      RefuelAll, RefuelPartial) are AUTHORITATIVE at the instant they fire.
      Their handlers write fuel_main (and fuel_reservoir where provided)
      directly.

   2. Between journal events, Status.json polling maintains the live gauge
      via Section 8.

   3. These two paths do not conflict in practice. They write the same
      fields with the same values derived from the same underlying game
      state. The only race is that Status.json may be up to 0.5 s stale
      relative to a journal event; this is acceptable and does not violate
      Law 7 — the 100 ms combat alert budget applies to broadcast
      dispatch, not to gauge freshness.

   4. Threshold evaluation (Section 10) is triggered ONLY by journal
      handlers. Status.json writes are silent. Rationale: Status.json fires
      every 0.5 s whether values changed or not; threshold crossings must
      debounce on discrete event boundaries or the broadcast layer will
      spam.

10. THRESHOLD LOGIC
================================================================================

A. KB entries (omnicovas/knowledge_base/combat_mechanics.json):

   Verify (V7) or add:

      "fuel_low_threshold": {
        "value": 0.25,
        "unit": "fraction",
        "patch_verified": "4.3.1.0",
        "source": "Frontier in-game fuel warning parity",
        "last_updated": "2026-04-24",
        "confidence": "high",
        "needs_review": false,
        "_justification":
          "25% is the ED in-game low-fuel warning threshold. Matches FUEL_LOW."
      },
      "fuel_critical_threshold": {
        "value": 0.10,
        "unit": "fraction",
        "patch_verified": "4.3.1.0",
        "source": "Frontier in-game fuel critical warning parity",
        "last_updated": "2026-04-24",
        "confidence": "high",
        "needs_review": false,
        "_justification":
          "10% is the ED in-game critical-fuel warning threshold."
      }

   If already present with matching values, do not rewrite — only bump
   last_updated and patch_verified if they are older than 4.3.1.0.
   Bump _metadata.json kb_version and last_full_audit_date if you modify
   either entry.

B. Computation:

   The fraction evaluated against these thresholds is:

      fraction = fuel_main / fuel_capacity_main

   Not against the sum of main + reservoir, and not against capacity_main +
   capacity_reserve. Rationale: the reservoir is a ~1.14 t buffer that is
   always full as long as main > 0. Including it in the denominator skews
   the fraction and delays the warning. Frontier's own in-game warning
   tracks main-only. We match that behaviour for parity.

   If fuel_main is None or fuel_capacity_main is None or
   fuel_capacity_main <= 0.0: skip threshold evaluation for this tick. Do
   not broadcast. Do not log a warning.

C. Hysteresis (existing Week 7 behaviour — preserve):

   FUEL_LOW is broadcast once on downward crossing of 0.25. It is not
   re-broadcast on subsequent FuelMain decrements. It is re-armed when
   fuel_main / fuel_capacity_main rises back above 0.25 (after a refuel or
   fuel scoop).

   FUEL_CRITICAL behaves identically on 0.10.

   Preserve whatever crossing-state fields (e.g. _fuel_low_fired: bool,
   _fuel_critical_fired: bool) already exist in fuel.py. Do not rename
   them.
