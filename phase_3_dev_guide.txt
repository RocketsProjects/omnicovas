HOW TO USE THIS DOCUMENT

This document is your step-by-step guide through Phase 3 of OmniCOVAS
development. Phase 3 makes everything Pillar 1 broadcasts in Phase 2 visible
to the Commander. If Phase 1 was the concrete and Phase 2 was the first floor,
Phase 3 is the windows, the front door, and the path to the street — the
moment OmniCOVAS becomes something a Commander can actually use.

GOLDEN RULE: Every Pillar 1 broadcast laid down in Phase 2 must be observable
in the UI by end of Week 14. The Tauri shell, the bridge endpoints, and the
overlay are the load-bearing deliverables — get them right before reaching
for the P2 polish items. Phase 3 also opens the external-approval window:
the Frontier Developer application and the Inara whitelisting request both
go out at Week 14 close.

--------------------------------------------------------------------------------
ARCHITECTURE PATTERNS FOR PHASE 3
--------------------------------------------------------------------------------

Pattern 1 — WebSocket-First State (subscribe, never poll)
  - The UI subscribes to /ws/events and renders from the broadcast stream
  - Polling /state is the safety net only, not the primary path
  - Every broadcast type from Phase 2 is forwarded to connected WS clients
  - Subscriber registered once in main(); not per-component

Pattern 2 — Confirmation Gate UI (Law 1 enforcement at the surface)
  - Every advisory output passes through a ConfirmationGate component
  - Component renders the suggestion, the "Why?" affordance (Tier 3), and
    explicit Confirm / Decline controls
  - No UI control fires an in-game action implicitly — Law 1 is absolute
  - Phase 3 lands the framework; Phase 4+ populates Tier 3 content

Pattern 3 — Privacy-by-Default Toggles (Principle 7)
  - Every outbound data flow has an explicit toggle on the Privacy page
  - Every toggle defaults OFF
  - Toggle state persists to the encrypted config vault (Phase 1 DPAPI)
  - The Data Flow Transparency Map renders zero active flows in the
    Phase 3 baseline because no Phase 3 feature ships outbound data

Pattern 4 — Accessibility from Day One (Principle 1)
  - Semantic HTML before custom widgets — use <button>, <nav>, <main>, etc.
  - ARIA labels on every interactive element from first commit, not retrofit
  - Focus order is the tab order; visible focus indicator is mandatory
  - Color is never the only signal (icons + text accompany color cues)

Pattern 5 — Three Output Modes Router (Voice / Overlay / Both)
  - Phase 3 owns the Overlay path
  - Voice path is stubbed (Phase 3+; requires EDDI/VoiceAttack — Law 9 B)
  - Output mode is a per-feature setting in the Settings panel
  - Default for Phase 3: Overlay-only, all critical events enabled

Pattern 6 — Three-Path Onboarding (single wizard, three branches)
  - Easy Mode — preset selection + skip; under 60 seconds wall-clock
  - Custom Setup — guided wizard, 5 to 15 minutes
  - Power User — link directly to Settings + Privacy pages
  - First-run flag persists in config; wizard does not re-trigger after success

PART A — Frontend Architecture Decision
--------------------------------------------------------------------------------

[ ] Decide frontend approach: vanilla HTML/JS vs framework
    Why: The Phase 1 skeleton is vanilla. A framework (Svelte, Solid, Lit,
    plain Web Components) reduces hand-written DOM code but adds a build
    step, dependencies, and a security-review surface. Document the choice
    before adding any framework code.
    Note: Decision goes in docs/decisions/0001-frontend-framework.md.
    Default if undecided: stay vanilla and use Web Components for reusable
    pieces. Vanilla is the lowest-risk path to a working prototype.

[ ] Establish design tokens
    Why: A single source of truth for color, spacing, typography prevents
    drift across pages and makes the color-blind palette swap (Week 14)
    a token-level change, not a per-file edit.
    Note: ui/styles/tokens.css. Define --color-bg, --color-fg, --color-accent,
    --color-warn, --color-critical, --space-*, --font-*. Every new component
    references tokens, never hard-coded values.

[ ] Build navigation shell (sidebar + content area)
    Why: Every Phase 3 page lives inside this shell. Stable navigation is a
    prerequisite for keyboard-only navigation in Week 14.
    Note: Routes for /dashboard (default), /activity-log, /settings,
    /privacy, /resources, /credits. Single keyboard handler at the shell
    level. <nav> with <ul> of <a> elements for screen-reader compatibility.

[ ] Build base layout primitives
    Why: Cards, lists, tables, badges, status dots reused across every view.
    Define them once.
    Note: ui/components/ — card.html (or .js), badge.html, status-dot.html,
    field-row.html. Each primitive accepts ARIA attributes via attribute
    pass-through.

PART B — FastAPI Bridge Extension (Pillar 1 Endpoints)
--------------------------------------------------------------------------------

NOTE: The Phase 1 bridge exposes /health, /state, /activity-log, /resources.
Phase 2 added /fuel, /jump_range, /rebuy. Phase 3 adds focused per-view
endpoints so each UI page fetches exactly what it needs and no more.

[ ] Create omnicovas/api/pillar1.py
    Why: Centralizes the new Phase 3 endpoints. Keeps api_bridge.py from
    growing unboundedly.
    Note: FastAPI APIRouter pattern. Mounted under prefix /pillar1. Tests
    live in tests/test_api_pillar1.py.

[ ] Add GET /pillar1/ship-state
    Why: Dashboard view needs the consolidated ship snapshot in one call.
    Note: Returns ship_type, ship_name, ship_ident, hull_health, shield_up,
    shield_strength_pct, fuel_main, fuel_capacity, jump_range_ly,
    cargo_count, cargo_capacity, heat_level, sys_pips, eng_pips, wep_pips,
    is_docked, current_system, current_station, is_wanted_in_system.

[ ] Add GET /pillar1/loadout
    Why: Loadout view needs the full module list with engineering blocks.
    Note: Returns ship_type, loadout_hash, module list (slot, item, health,
    power, priority, on, engineering raw block). Engineering remains raw —
    Pillar 6 (Phase 8) owns effect computation.

[ ] Add GET /pillar1/cargo
    Why: Cargo view filters and sorts the inventory; needs structured data.
    Note: Returns cargo_count, cargo_capacity, inventory list (name,
    localised_name, count, stolen, mission_id).

[ ] Add GET /pillar1/heat and GET /pillar1/pips
    Why: Compact endpoints for the dashboard cards and overlay banner.
    Note: heat returns level, trend (rising/steady/falling), last 10
    samples for sparkline. pips returns sys/eng/wep + last_changed_at.

[ ] Extend /ws/events to forward all Phase 2 broadcast types
    Why: WebSocket-First Pattern (Pattern 1). Subscribers must see every
    event type without registering for each one.
    Note: Single subscriber registered in main() that wraps every event
    type into the existing WebSocketBroadcaster.broadcast() call. Payload
    shape: {type: 'event', event_type, timestamp, payload, source}.

[ ] Write pytest tests for new endpoints
    Why: Endpoint shape is the UI contract. Snapshot it with tests.
    Note: tests/test_api_pillar1.py. Each endpoint: 200 happy path,
    correct shape, correct values from a seeded StateManager.

PART A — Tauri Overlay Window Configuration
--------------------------------------------------------------------------------

[ ] Add a second window definition to src-tauri/tauri.conf.json
    Why: Main shell and overlay are separate windows with different
    properties.
    Note: label: "overlay", transparent: true, alwaysOnTop: true,

    Why: Overlay has a separate document tree to keep its DOM minimal —
    transparency rendering is sensitive to large compositions.
    Note: Background must be fully transparent. No body padding. No

[ ] Add Tauri plugin for window-state persistence
    Why: Commanders expect the overlay to remember where they put it.
    Note: tauri-plugin-window-state is the canonical choice. License

    docs/decisions/0002-tauri-plugins.md.

[ ] Confirm transparency renders correctly on Windows 10 and 11
    Why: WebView2 transparency has historical edge cases.
    Note: window-vibrancy is already in src-tauri/Cargo.lock (verified
    via Phase 1). Test on a Windows 10 box and a Windows 11 box if
    available. Failure mode is a black or white frame instead of
    transparency — log and escalate, do not ship.

--------------------------------------------------------------------------------
PART B — Click-Through Behavior
--------------------------------------------------------------------------------

[ ] Implement click-through default
    Why: A Commander flying a ship must be able to click through the
    overlay to the game. Anything else is a Law 6 (Performance Priority)
    violation by way of disrupting flow.
    Note: Tauri set_ignore_cursor_events(true) on overlay window at
    creation. The overlay never steals input by default.

[ ] Implement hotkey to toggle click-through
    Why: When the Commander needs to dismiss a banner or interact with
    the overlay, they need a way to grab it.
    Note: Default Ctrl+Shift+O. Configurable in Settings (Week 13).
    Toggle is global hotkey via tauri-plugin-global-shortcut. License
    review before adding.

[ ] Visual indicator of click-through state
    Why: Hidden state is a foot-gun. Commanders must see whether the
    overlay is currently grabbing input.
    Note: Small status dot in the corner of the overlay. Different
    color for grab vs click-through. Disappears after 3 seconds when
    no banner is active.

--------------------------------------------------------------------------------
PART C — Critical Event Banner
--------------------------------------------------------------------------------

[ ] Subscribe to critical event types
    Why: The overlay's primary job is surfacing critical events without
    forcing the Commander to open the main shell.
    Note: HULL_CRITICAL_25, HULL_CRITICAL_10, SHIELDS_DOWN, FUEL_LOW,
    FUEL_CRITICAL, MODULE_CRITICAL, HEAT_WARNING. Subscriber registered
    in the overlay webview, separate from the main shell subscriber.

[ ] Build the banner component
    Why: Single visual primitive used by every critical event type.
    Note: Renders icon, label, value (e.g. "HULL 8%"), short context.
    Color follows the event severity. Token-driven so color-blind
    palette swap (Week 14) works without rewriting the banner.

[ ] Implement auto-dismiss timer (configurable per event type)
    Why: Banners that linger become noise.
    Note: Defaults: HULL_CRITICAL_10 = 30 s, HULL_CRITICAL_25 = 10 s,
    SHIELDS_DOWN = 5 s, FUEL_CRITICAL = 30 s, FUEL_LOW = 10 s,
    MODULE_CRITICAL = 15 s, HEAT_WARNING = 8 s. All configurable in
    Settings (Week 13).

[ ] Implement banner queue (FIFO with priority preemption)
    Why: A Commander hit by hull damage AND shield down at the same
    moment must see both, not just the first.
    Note: Priority order in Appendix. HULL_CRITICAL_10 preempts
    everything below it. Lower-priority banners are queued, not
    discarded; they show after the preempting banner clears.

PART A — First-Run Onboarding (Three-Path Model)
--------------------------------------------------------------------------------

[ ] Detect first run
    Why: First-Run only fires once per installation.
    Note: Check encrypted config for first_run_completed flag. If absent,
    show wizard; on completion, set flag.

[ ] Build the path-selection screen
    Why: A Commander chooses Easy, Custom, or Power User up front.
    Note: Three large buttons with one-line descriptions. Skip option
    available (treated as Easy Mode default).

[ ] Path A — Easy Mode (60-second setup)
    Why: Universal Pattern §10.B.1. Casual Pilot preset, NullProvider,
    privacy defaults at OFF, overlay enabled with critical events only.
    Note: One screen: confirm preset (Combat / Explorer / Trader / Miner /
    Casual). Hit Continue. Done. Wall-clock target: under 60 seconds.

[ ] Path B — Custom Setup (5 to 15 minute wizard)
    Why: Commanders who want some control without diving into every setting.
    Note: Multi-step wizard: preset, AI provider choice (NullProvider /
    Gemini / Local LLM / skip), privacy toggles (one screen, all OFF
    by default), output mode (Voice/Overlay/Both — Voice greyed out
    in Phase 3), overlay position. Each step has a Skip option.

[ ] Path C — Power User (link to full Settings + Privacy pages)
    Why: Some Commanders want to read every setting before any default
    is applied.
    Note: One screen with two big buttons: "Open Settings" and "Open
    Privacy Page". Wizard exits. first_run_completed is set to true
    only after the Commander has opened both at least once.

[ ] Persist first_run_completed flag
    Why: Wizard never re-triggers after success.
    Note: Encrypted config (DPAPI). Manual reset path: delete the config
    key via dev-tools menu (gated behind --dev flag).

--------------------------------------------------------------------------------
PART B — Privacy Page with Opt-In Toggles
--------------------------------------------------------------------------------

[ ] Build /privacy route
    Why: Single canonical location for every outbound-data toggle.
    Note: Heading: "Your data, your choice." Below it: a list of toggles,
    every one defaulted OFF.

[ ] Implement Phase 3 toggle inventory
    Why: Privacy-by-Default Pattern. Every potential outbound flow has
    a toggle, even if no Phase 3 feature uses it yet.
    Note: Toggles to render:
      - EDDN market data submission (Phase 5+)
      - EDSM commander tracking (Phase 5+)
      - Squadron telemetry sync (Phase 7)
      - AI provider API calls (Tier 3 features, Phase 4+)
      - Crash report submission (currently inert; structural)
      - Usage analytics (currently none planned; structural)
    Each toggle has a one-paragraph plain-language description.

[ ] All toggles default OFF; setting persists to encrypted vault
    Why: Principle 7 (Privacy-by-Default). Law 8 (Sovereignty &
    Transparency).
    Note: No exceptions, ever. The Commander turns each toggle on if
    they want it on. The system never assumes consent.

[ ] Implement export and delete capability
    Why: GDPR alignment (Compliance Matrix Section 5.1).
    Note: "Export my data" — produces a JSON dump of every config + KB
    state + recent Activity Log entries. Saved via OS file dialog.
    "Delete all my data" — confirms twice, then wipes the config vault
    + activity log + cached KB. Restart required after delete.

PART D — Settings Panel (Three-Tier Customization)
--------------------------------------------------------------------------------

[ ] Tier 1 — Preset Profiles
    Why: Universal Pattern §10.C.1. The fastest path to a sensible config.

[ ] Tier 2 — Category Toggles
    Why: Universal Pattern §10.C.2. Whole-category enable / disable.
    Note: Categories: Ship Telemetry (Pillar 1), Combat (Pillar 2 —
    greyed out until Phase 4), Exploration (Phase 5), Trading (Phase 6),

[ ] Tier 3 — Granular Settings
    Why: Universal Pattern §10.C.3. Every feature has an explicit control.

PART A — Accessibility Framework (Principle 1, P1)
--------------------------------------------------------------------------------

[ ] Audit semantic HTML across every Phase 3 view
    Why: Pattern 4. Screen readers and keyboard-only users depend on
    semantic structure.
    Note: Replace any <div role="button"> with <button>. Replace any
    <div class="nav"> with <nav>. <main>, <section>, <h1>-<h3>, <ul>
    used correctly. One <h1> per view.

[ ] ARIA labels on every interactive element
    Why: Even semantic HTML benefits from explicit labels for icon-only
    controls and dynamic regions.
    Note: aria-label on icon buttons. aria-live="polite" on the
    Activity Log live region. aria-live="assertive" on the overlay
    critical banner. aria-describedby for tooltips.

[ ] Focus order audit (logical tab order)
    Why: Tab order should follow reading order without surprises.
    Note: Walk every view with Tab key. Document any non-source-order
    placements that require tabindex. Default tabindex of 0 for custom
    interactive elements.

[ ] Visible focus indicators (Tier 1 mandatory)
    Why: Keyboard users must see where they are.
    Note: :focus-visible outlines using --color-accent token. No
    outline: none anywhere unless replaced with a more visible style.

[ ] Screen reader smoke test (NVDA on Windows)
    Why: NVDA is free and the most common Windows screen reader.
    Note: Walk Dashboard, Privacy, Settings, Activity Log, Onboarding.
    Confirm every label is announced. Confirm critical banner is

SEVEN-LAYER UI DEBUGGING VOCABULARY
--------------------------------------------------------------------------------

Trace in this order. UI bugs hide in transitions between layers.

1. Bridge endpoint        Is the FastAPI route returning correct data?
2. WebSocket              Is the broadcast forwarded to /ws/events?
3. Tauri window           Is the webview loaded and the route correct?
4. HTML / Component       Is the DOM rendered? Are tokens applied?
5. State hook             Is the WS message updating the component?
6. Style / Token          Is the visual change wired to a token? (palette
                          and font-size paths depend on this)
7. Accessibility surface  Is the change announced to screen readers?
                          Is focus order preserved?
